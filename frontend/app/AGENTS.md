# Codex 规则

## 基本原则
- 通用工作方式与 Vue 基线规范以 `.codex` 为主。
- 本文件仅保留 uni-app 商城端的目录、接口、状态、样式、多端兼容和校验命令等差异规则。

## 项目概览
- 当前项目技术栈为 `uni-app + Vue 3 + TypeScript + Vite + Pinia + Sass`。
- 页面按主包与分包组织，主包目录为 `src/pages`，会员分包为 `src/pagesMember`，订单分包为 `src/pagesOrder`。
- 接口调用代码位于 `src/api`，接口与数据模型类型主要来自 `src/rpc`。
- 全局状态集中在 `src/stores/modules`，通用请求封装位于 `src/utils/http.ts`。

## 目录与职责约定
- 新增页面时，优先放入与业务最接近的包目录：主包放 `src/pages`，会员相关放 `src/pagesMember`，订单相关放 `src/pagesOrder`。
- 新增页面后，必须同步更新 `src/pages.json`，保持路由、分包、导航配置一致。
- 页面私有组件优先放在页面目录下的 `components` 子目录，例如 `src/pages/index/components`。
- 通用组件放在 `src/components`，只有在至少两个页面或模块复用时，才提升为全局组件。
- 通用组合式逻辑放在 `src/composables`，不要把跨页面复用逻辑散落在页面 `.vue` 文件中。
- 工具方法放在 `src/utils`，状态管理只放在 `src/stores`，不要让页面直接承担全局状态职责。

## 页面开发约定
- 页面默认使用 `script setup` + TypeScript，保持当前仓库的实现风格。
- 页面数据、事件、请求逻辑按“状态定义 -> 计算属性 -> 数据加载 -> 事件方法”组织，避免模板中塞入复杂表达式。
- 页面初始化请求优先收敛为 `loadData`、`getXxxData` 这类语义明确的方法，不要把多个接口直接堆在生命周期钩子里。
- 下拉刷新、触底加载、分享配置、弹窗开关等高频交互，优先延续现有页面的拆分方式，保持方法粒度清晰。
- 需要登录态才能访问的页面操作，统一先检查 `useUserStore()` 中的用户信息，再决定跳转登录页或继续执行。
- 页面跳转时，URL 参数名保持语义化，已有页面已使用的参数名不要随意改写，避免破坏路由兼容。

## 接口与类型约定
- 所有业务请求统一通过 `src/api` 下的 service 类发起，不要在页面、store、组件中直接手写 `uni.request`。
- 通用请求能力、鉴权、刷新 token、错误提示统一复用 `src/utils/http.ts`，不要重复实现第二套请求封装。
- 页面、组件、store 使用的业务类型，优先从 `src/rpc` 中导入，不要在业务代码里重复声明等价类型。
- `src/rpc` 下的文件默认视为生成产物或协议产物；除非明确是在修正生成结果或协议定义，否则不要随意手改。
- 新增业务接口时，先补充 `src/api` 中对应 service，再在页面或 store 中调用，不要跳过 service 这一层。
- 接口命名、service 命名、导出实例命名优先沿用现有模式，例如 `XxxServiceImpl` 与 `defXxxService`。

## Store 与登录态约定
- 全局共享状态统一放入 Pinia store，页面局部临时状态继续放在页面内的 `ref` 或 `computed` 中。
- store 模块继续放在 `src/stores/modules`，并在 `src/stores/index.ts` 统一导出。
- Token、刷新 token、过期时间的读写统一走 `src/utils/auth.ts`，不要在别处直接读写同名存储键。
- 涉及登录失效、刷新 token、重新登录跳转的逻辑，优先复用当前 `src/utils/http.ts` 机制，不要在页面重复处理。

## 样式与组件约定
- 全局基础样式优先复用 `src/styles/base.scss`、`src/styles/fonts.scss`、`src/uni.scss` 中已有约定。
- 页面样式优先使用当前项目已存在的色值、间距和结构写法，避免单页独立发明一套视觉规则。
- 组件命名延续现有风格：全局组件使用 `Xtx` 前缀，页面私有组件采用业务语义命名。
- 已通过 `pages.json` easycom 配置自动引入的组件，不要重复手写可省略的全局注册逻辑。
- 样式修改应优先保证微信小程序端可用，同时兼顾 H5；涉及平台差异时，明确使用条件编译。

## 条件编译与多端兼容约定
- 涉及平台差异的代码必须显式使用 `#ifdef`、`#ifndef` 等条件编译标记，避免把平台判断写成运行时分支散落在业务代码中。
- 修改登录、路由、存储、分享、支付、预览图片等平台敏感逻辑时，必须同时检查 `MP-WEIXIN`、`H5`、`APP-PLUS` 的兼容性。
- 如果某项能力仅支持单端，代码与注释都要明确说明，不要让其他端静默失败。

## 变更边界约定
- 修改页面功能时，优先在页面、私有组件、service、store 这些明确边界内完成；不要把业务逻辑无序扩散到全局工具层。
- 若改动会影响路由结构、分包结构、请求封装、登录态机制、全局 store 或通用组件接口，先梳理影响范围后再动手。

## 注释与可读性约定
- 新增或修改关键逻辑时，补充简洁中文注释，重点说明副作用、边界条件、平台差异和业务原因。
- 公共方法、复杂异步流程、登录态分支、条件编译分支需要注释；明显的模板映射或简单赋值不强制加注释。
- 不要为了“满足注释数量”写低价值注释，注释必须帮助后来者理解代码。

## 校验与文档约定
- 修改代码文件后，至少执行与改动匹配的检查。
- 当前项目默认检查命令为 `pnpm lint` 与 `pnpm tsc`。
- 若因仓库历史问题导致无法全量通过，必须在结论中说明执行了什么、失败在哪里、失败是否由本次改动引起。
- 新增页面、目录调整、构建方式变化、环境变量变化、核心开发约定变化后，需同步更新 `README.md`。
- 若发现 README 与当前代码结构不一致，后续相关任务中应优先顺手修正，避免文档持续失真。
