# 后台首页与数据分析整合改版方案

## 1. 背景

当前后台首页相关能力分成两块：

- 工作台： [workspace/index.vue](/Users/liujun/workspace/shop/shop/frontend/admin/src/views/dashboard/workspace/index.vue)
- 数据分析：  
  - [analytics/user/index.vue](/Users/liujun/workspace/shop/shop/frontend/admin/src/views/dashboard/analytics/user/index.vue)  
  - [analytics/goods/index.vue](/Users/liujun/workspace/shop/shop/frontend/admin/src/views/dashboard/analytics/goods/index.vue)  
  - [analytics/order/index.vue](/Users/liujun/workspace/shop/shop/frontend/admin/src/views/dashboard/analytics/order/index.vue)

当前问题不是“页面数量多”，而是 **首页层级不清**：

- 工作台偏静态，缺少真实业务数据。
- 数据分析已经有用户、商品、订单三套能力，但和首页没有形成联动。
- 用户进入后台后，既不知道先处理什么，也不知道去哪看原因。

所以本次整合的目标，不是把工作台和分析页做成一个页面，而是重新定义两者边界，并建立完整跳转链路。

## 2. 整合目标

整合后，首页体系分成两层：

### 2.1 工作台

定位：**运营值班台**

回答的问题：

- 今天先处理什么
- 当前哪里有积压
- 哪些模块异常
- 点哪里能直接去处理

### 2.2 数据分析

定位：**经营分析页**

回答的问题：

- 用户为什么变了
- 商品结构哪里有问题
- 订单和销售额趋势怎么变化
- 哪个维度贡献变化最大

一句话说：

- 工作台看“现在做什么”
- 数据分析看“为什么这样”

## 3. 当前已具备的分析能力

后端已经拆出三套分析服务：

- [user_analytics.proto](/Users/liujun/workspace/shop/shop/backend/api/protos/admin/user_analytics.proto)
- [goods_analytics.proto](/Users/liujun/workspace/shop/shop/backend/api/protos/admin/goods_analytics.proto)
- [order_analytics.proto](/Users/liujun/workspace/shop/shop/backend/api/protos/admin/order_analytics.proto)

分别提供：

### 3.1 用户分析

- 摘要：新增用户、增长率、下单用户、转化占比、活跃行为用户、覆盖用户
- 趋势：注册用户 vs 下单用户
- 行为覆盖排行：地址、收藏、加购、下单、门店申请

### 3.2 商品分析

- 摘要：新增商品、上架完成率、动销商品、动销率、商品销量、较上期
- 趋势：销量 vs 动销商品数
- 分布：分类销量分布

### 3.3 订单分析

- 摘要：新增订单、增长率、成交金额、客单价、下单用户、复购占比
- 趋势：订单量 vs 销售额
- 分布：订单状态分布

这些能力不应该再重复做一份到工作台，而应该作为工作台的“下钻层”。

## 4. 整合后的首页结构

建议后台首页保留一级菜单“首页”，下含两个入口：

1. 工作台
2. 数据分析

其中数据分析继续保留三页签：

- 用户分析
- 商品分析
- 订单分析

这样改的好处：

- 路由层级清楚
- 工作台不被趋势图挤满
- 分析页不被待办项打断

## 5. 新的信息架构

### 5.1 工作台页面结构

建议拆成 4 个区域：

#### A. 顶部总览

只放“最该先看”的 4 张卡：

- 待发货订单
- 今日成交额
- 动销商品
- 待审核门店

这 4 张卡全部来自已有表结构：

- `order`
- `order_goods`
- `goods`
- `user_store`

每张卡都要提供两个动作：

- 点击卡片进入对应业务页
- 次级链接“查看分析”

例如：

- 待发货订单 -> `/order/info`
- 查看分析 -> `/dashboard/analytics/order`

#### B. 左主区：待处理事项

这一区是工作台核心。

建议 4 张业务动作卡：

1. 订单履约
2. 退款处理
3. 商品巡检
4. 门店审核

每张卡只展示：

- 当前积压量
- 今日新增量
- 一条风险提示
- 一个主按钮

这里不做趋势图，不做大面积统计。

#### C. 右侧：分析快照

这一区是“工作台和分析页的接缝”。

不做完整图表，只做轻量摘要，点击后跳分析页。

建议三张小卡：

1. 用户快照
   - 今日新增用户
   - 下单用户
   - 活跃行为用户
   - 跳转用户分析

2. 商品快照
   - 新增商品
   - 动销商品
   - 商品销量
   - 跳转商品分析

3. 订单快照
   - 新增订单
   - 成交金额
   - 客单价
   - 跳转订单分析

注意：这里不要再重复整页分析卡片，只保留“摘要 + 入口”。

#### D. 底部：运营位与快捷入口

继续保留，但聚焦业务：

- 轮播图
- 热门推荐
- 服务项配置
- 商品管理
- 订单管理
- 门店审核

## 6. 数据分析页面结构重定义

当前分析页已经拆成用户、商品、订单三页，这是正确方向。  
整合后不建议再做“经营数据总览页”，而是明确这三页的职责。

### 6.1 用户分析页

定位：看增长与行为沉淀。

保留模块：

- 摘要卡：新增用户、下单用户、活跃行为用户
- 趋势图：注册用户 vs 下单用户
- 行为覆盖图：地址 / 收藏 / 加购 / 下单 / 门店申请

建议补一个说明：

- 当前“转化占比”和“覆盖用户”分母是当前周期新增用户，不是全量用户

### 6.2 商品分析页

定位：看供给规模与动销效率。

保留模块：

- 摘要卡：新增商品、动销商品、商品销量
- 趋势图：销量 vs 动销商品数
- 分类分布：分类销量结构

建议后续增加：

- 下架商品数
- 无销量商品数

### 6.3 订单分析页

定位：看交易结果与履约结构。

保留模块：

- 摘要卡：新增订单、成交金额、下单用户
- 趋势图：订单量 vs 销售额
- 状态分布：待发货 / 待收货 / 已完成 / 已退款 / 已取消

建议后续增加：

- 退款订单占比
- 取消订单占比

## 7. 首页与分析页的联动规则

这是这次整合最关键的一点。

### 7.1 工作台的摘要卡必须能跳分析页

建议：

- 用户快照卡 -> 用户分析页
- 商品快照卡 -> 商品分析页
- 订单快照卡 -> 订单分析页

### 7.2 工作台的待办卡必须跳业务处理页

建议：

- 订单履约 -> 订单管理
- 商品巡检 -> 商品管理
- 门店审核 -> 门店审核页
- 运营位巡检 -> 轮播图 / 热门推荐 / 服务项

### 7.3 分析页顶部增加“回到工作台”或“去处理”入口

例如订单分析页：

- 查看订单管理
- 回到工作台

这样工作流闭环才完整：

工作台发现问题 -> 去分析页看原因 -> 去业务页处理

## 8. 推荐的页面布局方案

### 8.1 工作台

```text
+--------------------------------------------------------------+
| 顶部总览：待发货 / 今日成交额 / 动销商品 / 待审核门店          |
+--------------------------------------+-----------------------+
| 待处理事项                           | 分析快照              |
| - 订单履约                           | - 用户快照            |
| - 退款处理                           | - 商品快照            |
| - 商品巡检                           | - 订单快照            |
| - 门店审核                           |                       |
+--------------------------------------+-----------------------+
| 运营位巡检                           | 快捷入口              |
+--------------------------------------+-----------------------+
```

### 8.2 分析页

分析页保持当前结构：

- 顶部摘要卡
- 左侧主图
- 右侧辅助图

但统一交互规范：

- 顶部有时间切换
- 摘要卡只放摘要
- 图表卡底部留一个“查看业务页”入口

## 9. 后端接口建议

### 9.1 保留现有分析接口

现有三套分析接口已经足够支撑分析页，不建议推翻。

继续保留：

- `GetUserAnalyticsSummary/Trend/Rank`
- `GetGoodsAnalyticsSummary/Trend/Pie`
- `GetOrderAnalyticsSummary/Trend/Pie`

### 9.2 新增工作台聚合接口

工作台单独设计一套聚合服务，不复用分析页接口。

建议增加：

- `GetWorkspaceOverview`
- `GetWorkspaceTodo`
- `GetWorkspaceSnapshot`
- `GetWorkspaceOperationHealth`

理由：

- 工作台要的是“当前状态聚合”
- 分析页要的是“时间维度统计”
- 两者口径不一样

## 10. 菜单与路由建议

建议最终结构：

- 首页
  - 工作台
  - 数据分析
    - 用户分析
    - 商品分析
    - 订单分析

不建议：

- 再保留旧的“经营数据总览页”
- 工作台里塞完整趋势图
- 分析页里混入待办事项

## 11. 实施优先级

### 第一阶段

只改结构，不大改视觉：

- 工作台换成“总览 + 待办 + 快照 + 入口”
- 分析页保留三页结构
- 补工作台到分析页、分析页到业务页的跳转

### 第二阶段

补工作台聚合接口。

### 第三阶段

补运营位健康、对账异常、退款异常等高级提醒。

## 12. 最终结论

当前已有的数据分析能力不应该被“整合掉”，而应该被 **重新放到正确位置**：

- 工作台承接“发现问题”和“安排动作”
- 数据分析承接“解释原因”和“看趋势结构”
- 业务页承接“真正处理”

所以这次重新设计的核心不是页面合并，而是首页体系重排：

1. 工作台做成运营值班台
2. 分析页继续保持用户 / 商品 / 订单三拆分
3. 两者通过摘要快照和处理入口打通

这比单纯把工作台做成“更大的一张看板”更符合当前项目的数据基础和后台场景。
