# Gorse 推荐服务管理平台功能设计

## 1. 文档目标

本文档用于设计“管理后台如何承接 Gorse 推荐服务的管理能力”。目标不是直接替换现有推荐链路，而是在现有商城推荐能力基础上，为运营、产品、技术和运维提供可视化、可配置、可调试、可发布的后台入口。

本文档只讨论管理后台能力设计，不涉及本次直接开发实现。

## 2. 现状说明

结合当前仓库实现，推荐体系已经具备以下基础能力：

- App 侧已经接入 Gorse 推荐链路，支持首页、商品详情、购物车、个人中心、订单详情、支付成功页等推荐场景。
- 后端已经支持推荐事件上报，事件类型包含曝光、点击、浏览、收藏、加购、下单、支付。
- 后端已经支持 Gorse 推荐服务主数据同步，包括用户、商品全量同步和异步增量同步。
- 后端已经支持历史推荐事件回放到 Gorse 推荐服务。
- 在线推荐请求已经具备责任链和本地兜底逻辑，Gorse 不可用时可以退回本地推荐。
- 管理后台当前仅覆盖“热门推荐”这类人工运营推荐，没有覆盖 Gorse 推荐服务的运维、调试、策略管理和发布能力。

这意味着当前系统“推荐能力已经存在”，但“后台管理能力明显缺位”。

## 3. 设计目标

### 3.1 目标

- 让后台能够看见 Gorse 推荐服务 是否可用，而不是只能依赖日志排查。
- 让后台能够管理推荐场景策略，而不是把全部规则固化在代码与配置文件中。
- 让后台能够管理同步、回放、调试、发布、回滚、告警等日常运维动作。
- 让运营和产品能够理解“当前每个场景用的是什么推荐逻辑、效果如何、是否发生兜底”。

### 3.2 非目标

- 一期不建议直接在后台暴露 `config.toml` 全量在线编辑，避免误操作导致推荐服务不可用。
- 一期不替换现有“热门推荐”页面，两者定位不同，应并存。
- 一期不在后台承担 Gorse 推荐服务 部署、容器编排、训练节点运维等基础设施职责。

## 4. 后台模块定位

建议在现有“推荐管理”下拆成两类能力：

- `人工运营推荐`
  - 继续保留现有“热门推荐”能力，适合固定坑位、专题位、人工编排内容。
- `Gorse 推荐服务`
  - 负责算法推荐的接入状态、场景策略、同步任务、反馈事件、调试验证、版本发布、监控告警。

建议后台菜单结构如下：

```text
推荐管理
├── 热门推荐
└── Gorse 推荐服务
    ├── 运行总览
    ├── 场景策略
    ├── 数据同步
    ├── 反馈事件
    ├── 推荐调试
    ├── 版本发布
    └── 告警中心
```

## 5. 功能范围总览

| 模块 | 能做什么 | 使用角色 | 优先级 |
| --- | --- | --- | --- |
| 运行总览 | 查看 Gorse 推荐服务 连通性、请求量、命中率、兜底率、异常率、最近同步状态 | 技术、运维、产品 | P0 |
| 场景策略 | 管理每个推荐场景的启用状态、责任链、兜底规则、分页参数、发布版本 | 产品、技术 | P1 |
| 数据同步 | 用户/商品全量同步、单对象补同步、历史事件回放、失败重试 | 技术、运维 | P0 |
| 反馈事件 | 查看事件映射、事件量、失败事件、事件回放记录 | 技术、产品 | P0 |
| 推荐调试 | 用指定主体、商品、订单做在线调试，查看链路轨迹和最终结果 | 技术、测试、产品 | P0 |
| 版本发布 | 策略草稿、发布、回滚、灰度说明、发布记录 | 产品、技术 | P1 |
| 告警中心 | 配置告警阈值，查看异常记录和恢复情况 | 技术、运维 | P1 |
| 实验分析 | 场景对比、版本效果对比、灰度效果评估 | 产品、运营、技术 | P2 |

## 6. 详细功能设计

### 6.1 运行总览

#### 页面目标

回答三个问题：

- Gorse 推荐服务 现在能不能用？
- 当前推荐请求主要走Gorse还是本地兜底？
- 最近有没有同步、回放、训练或接口异常？

#### 页面内容

1. 顶部指标卡

- Gorse 推荐服务 连接状态
- 最近 24 小时推荐请求数
- Gorse 命中率
- 本地兜底率
- 请求失败率
- 平均响应时长
- 最近一次全量同步时间
- 最近一次事件回放时间

2. 趋势图

- 按场景统计推荐请求趋势
- 按场景统计Gorse 命中率趋势
- 按场景统计本地兜底率趋势
- 按推荐提供方统计命中分布

3. 异常面板

- 最近失败的同步任务
- 最近失败的事件回放任务
- 最近失败的推荐调试任务
- 最近接口异常摘要

4. 快捷操作

- 测试 Gorse 推荐服务 连通性
- 触发用户全量同步
- 触发商品全量同步
- 新建事件回放任务
- 进入推荐调试页

#### 一期实现建议

- 页面优先做“可观测”，主要展示后端已有的请求、同步、回放结果。
- 对 Gorse 推荐服务 节点、模型训练、缓存状态等更底层指标，先做摘要，不必一开始铺太细。

### 6.2 场景策略

#### 页面目标

管理“什么场景使用什么推荐逻辑”，并为后续版本发布和回滚提供配置基础。

#### 当前代码基础

当前系统已经有明确场景：

- 首页
- 商品详情
- 购物车
- 个人中心
- 订单详情
- 支付成功页

当前 Gorse 推荐也已经存在明确责任链，例如：

- 首页：个性化推荐 -> 相似用户 -> 会话推荐 -> 30 天热门 -> 最新商品
- 商品详情：相邻商品 -> 商品关联 -> 会话推荐 -> 30 天热门 -> 最新商品
- 订单详情：商品关联 -> 相邻商品 -> 会话推荐 -> 支付热榜 -> 30 天热门 -> 最新商品

#### 页面设计

1. 场景列表

- 场景名称
- 场景编码
- 是否启用Gorse 推荐
- 当前发布版本
- 默认分页大小
- 当前责任链摘要
- 当前兜底策略摘要
- 最近发布时间
- 最近更新人

2. 场景编辑弹窗/详情页

- 基本信息
  - 场景名称
  - 场景编码
  - 场景说明
  - 是否启用
- Gorse 推荐配置
  - 是否启用Gorse 推荐
  - 推荐主体要求
  - 是否允许匿名
  - 默认页码/页大小
  - 责任链顺序
- 兜底配置
  - 同类目兜底是否启用
  - 最新热销兜底是否启用
  - 空结果处理策略
- 说明配置
  - 页面对业务方展示的场景说明
  - 调试备注

3. 版本动作

- 保存草稿
- 发布当前版本
- 回滚到历史版本
- 查看版本差异

#### 设计注意点

- 一期建议先把当前硬编码责任链“后台可读化”，即先支持查看和调试。
- 二期再将责任链、兜底规则从代码中抽离成后台配置，避免一次性改动过大。

### 6.3 数据同步

#### 页面目标

让后台能管理“推荐主数据是否已同步到 Gorse 推荐服务”。

#### 页面内容

1. 同步概览

- 用户总量
- 商品总量
- Gorse 推荐服务 用户量
- Gorse 推荐服务 商品量
- 差异数量
- 最近全量同步状态

2. 同步任务列表

- 任务编号
- 任务类型
  - 用户全量同步
  - 商品全量同步
  - 单用户补同步
  - 单商品补同步
  - 冗余数据清理
- 批次大小
- 发起方式
  - 手动
  - 定时
  - 异步增量
- 执行状态
- 成功数/失败数
- 开始时间/结束时间
- 操作人
- 失败原因

3. 操作能力

- 手动触发用户全量同步
- 手动触发商品全量同步
- 按用户 ID 补同步
- 按商品 ID 补同步
- 触发冗余数据清理
- 失败任务重试

#### 一期实现建议

- 优先补齐“任务记录”和“手动触发”。
- 对异步增量同步先展示成功/失败统计，不必一开始暴露过细队列参数。

### 6.4 反馈事件

#### 页面目标

让后台能知道“哪些行为正在喂给 Gorse 推荐服务，事件数据是否健康”。

#### 当前事件范围

- 曝光
- 点击
- 浏览
- 收藏
- 加购
- 下单
- 支付

#### 页面内容

1. 事件统计

- 按事件类型统计近 1 天 / 7 天 / 30 天数量
- 按场景统计事件量
- 按事件类型统计失败率

2. 事件映射说明

- 本地事件类型
- Gorse 推荐服务 反馈类型
- 权重说明
- 是否作为正反馈
- 是否作为读反馈

3. 回放任务列表

- 回放范围
- 回放时间区间
- 回放主体范围
- 回放状态
- 成功数
- 失败数
- 发起人
- 发起时间

4. 快捷操作

- 新建事件回放任务
- 重试失败回放任务
- 导出失败样本

#### 一期实现建议

- 一期先做“事件看板 + 历史回放任务”。
- 在线改权重不建议直接开放，可先做只读展示；真正要改权重，建议走版本化配置。

### 6.5 推荐调试

#### 页面目标

这是最关键的一页，用来回答“某个主体在某个场景下为什么拿到这批商品”。

#### 输入项

- 主体类型
  - 字典项：匿名 / 用户
- 主体信息
  - 根据主体类型联动选择或展示，不单独显示主体编号
- 推荐场景
  - 由推荐场景字典统一维护
- 商品 ID
- 订单 ID
- 上下文商品 ID 列表
- 页码
- 页大小

#### 输出项

1. 请求摘要

- 请求参数
- 请求编号
- 场景
- 主体信息
- 上下文商品

2. Gorse 推荐轨迹

- 责任链顺序
- 每个 provider 是否执行成功
- 每个 provider 返回数量
- 命中的 provider
- provider 错误信息

3. 最终结果

- 结果来源
  - Gorse 推荐服务
  - 本地兜底
- 兜底策略类型
- 返回商品列表
- 总数

4. 对比能力

- 对比“仅Gorse 结果”和“最终落地结果”
- 对比“当前发布版本”和“历史版本”

#### 页面价值

- 开发可快速排查“为什么没命中Gorse 推荐”。
- 测试可验证新场景配置是否生效。
- 产品可理解不同场景的推荐语义，不用直接读代码。

### 6.6 版本发布

#### 页面目标

将“场景策略、兜底规则、事件映射说明”等配置做成可发布版本，避免直接改线上生效。

#### 页面内容

- 版本编号
- 版本名称
- 状态
  - 草稿
  - 已发布
  - 已回滚
- 变更说明
- 生效范围
  - 全量
  - 指定场景
  - 灰度范围
- 发布时间
- 发布人

#### 页面动作

- 新建版本
- 复制已有版本
- 发布
- 回滚
- 查看版本差异
- 查看发布记录

#### 一期实现建议

- 一期先支持“草稿 / 发布 / 回滚”。
- 灰度发布、A/B 实验可以放到二期或三期。

### 6.7 告警中心

#### 页面目标

当Gorse 推荐不可用、兜底率异常、同步失败时，后台要能第一时间发现。

#### 告警规则建议

- Gorse 推荐服务 连续连接失败
- 某场景Gorse 命中率低于阈值
- 某场景本地兜底率高于阈值
- 事件回放失败数超过阈值
- 用户/商品同步失败数超过阈值
- 推荐请求平均耗时超过阈值

#### 页面内容

- 告警规则列表
- 告警记录列表
- 告警状态
  - 待处理
  - 已恢复
  - 已忽略
- 告警通知渠道说明

## 7. 推荐后台一期与后续分期

### 7.1 一期建议

一期优先解决“能看、能查、能调、能补”：

- 运行总览
- 数据同步
- 反馈事件
- 推荐调试
- 基础版本记录

### 7.2 二期建议

二期解决“能配、能发、能回滚”：

- 场景策略可编辑
- 责任链后台化
- 兜底规则后台化
- 正式版本发布与回滚
- 告警规则管理

### 7.3 三期建议

三期解决“能优化、能实验、能分析”：

- 灰度发布
- A/B 实验
- 版本效果对比
- 人群分层策略
- 推荐效果报表

## 8. 与现有热门推荐的关系

Gorse 推荐服务管理不是替代“热门推荐”页面，而是补齐另一类能力：

- 热门推荐：人工运营编排，适合固定专区、固定坑位、专题活动。
- Gorse 推荐服务：算法推荐分发，适合首页流、详情页相似商品、购物车猜你喜欢、订单页关联推荐等动态场景。

建议后台上将两者明确区分，避免运营误认为“热门推荐”已经覆盖算法推荐运维。

## 9. 数据与接口设计建议

### 9.1 推荐新增数据对象

建议后续根据分期考虑新增以下后台管理对象：

- `recommend_scene_strategy`
  - 场景策略定义
- `recommend_strategy_version`
  - 策略版本
- `recommend_strategy_publish_record`
  - 发布记录
- `recommend_sync_task`
  - 同步任务
- `recommend_event_replay_task`
  - 回放任务
- `recommend_alert_rule`
  - 告警规则
- `recommend_alert_record`
  - 告警记录
- `recommend_debug_snapshot`
  - 调试快照

### 9.2 管理后台接口建议

建议接口按 RESTful 风格设计，示例如下：

- `GET /api/admin/recommend/gorse/overview`
- `GET /api/admin/recommend/scene`
- `GET /api/admin/recommend/scene/{id}`
- `POST /api/admin/recommend/scene-version`
- `PUT /api/admin/recommend/scene-version/{id}/publish`
- `PUT /api/admin/recommend/scene-version/{id}/rollback`
- `GET /api/admin/recommend/sync-task`
- `POST /api/admin/recommend/sync-task`
- `POST /api/admin/recommend/event-replay-task`
- `GET /api/admin/recommend/event`
- `POST /api/admin/recommend/debug`
- `GET /api/admin/recommend/alert-record`

## 10. 权限建议

建议为后台按钮权限单独拆分，避免所有推荐能力共用一套粗粒度权限：

- `recommend:gorse:overview`
- `recommend:gorse:scene`
- `recommend:gorse:scene:update`
- `recommend:gorse:publish`
- `recommend:gorse:rollback`
- `recommend:gorse:sync`
- `recommend:gorse:event`
- `recommend:gorse:debug`
- `recommend:gorse:alert`

## 11. 风险与约束

- 当前 Gorse 推荐责任链存在代码硬编码，一期如果直接要求“后台改完立刻生效”，需要先做后端配置抽象。
- Gorse 推荐服务的推荐器、热榜公式、反馈权重目前与配置文件强相关，不建议一开始全部做成在线编辑。
- 若后台开放“回放、全量同步、回滚”能力，必须补齐操作审计和权限控制。
- 推荐结果存在本地兜底，后台指标必须同时区分“Gorse 命中”和“最终返回”，否则容易误判效果。

## 12. 结论

针对 Gorse 推荐服务，管理平台最值得优先建设的不是“在线改算法参数”，而是四类基础能力：

- 看得见：运行总览、请求来源、兜底率、异常率。
- 管得住：同步任务、事件回放、版本发布、回滚。
- 查得清：推荐调试、责任链轨迹、结果来源。
- 能迭代：场景策略配置化、灰度实验、效果分析。

如果按投入产出比排序，建议优先落地：

1. 运行总览
2. 数据同步
3. 反馈事件
4. 推荐调试
5. 场景策略
6. 版本发布
7. 告警中心

这样既能快速补齐当前后台缺口，也能为后续把推荐逻辑从“代码硬编码”演进到“后台版本化管理”打基础。
