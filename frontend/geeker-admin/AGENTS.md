# Codex 规则

## 适用范围
- 本规则适用于当前仓库根目录及其全部子目录。

## 页面开发约定
- 后续新增或修改后台列表页时，优先参考 `src/views/shop/banner/index.vue` 的实现方式，保持“`ProTable + FormDialog + ProForm`”的统一结构。
- 列表页优先使用 `ProTable` 的配置化能力，不要在页面里重复堆砌大量具名插槽；能通过 `columns`、`headerActions`、`cellType`、`actions` 表达的内容，统一走配置。
- 顶部按钮优先使用 `headerActions` 配置；行内按钮优先使用 `actions` 配置。仅在现有配置能力无法覆盖时，才退回页面 slot 或自定义 render。
- 图片列优先使用 `cellType: "image"`；状态列优先使用 `cellType: "status"`；不要在业务页面重复手写同类 `el-image`、`el-switch` 模板。
- 列表页弹窗优先使用 `src/components/Dialog/FormDialog.vue`；仅当表单结构明显不适合 `ProForm` 时，才直接使用 `ProDialog` 或 `el-dialog`。
- 后续 `el-form` 场景默认按可迁移到 `ProForm` 的方向编写，避免新增难以收敛的原生表单模式。
- 新增或修改业务功能后，必须同步检查并更新 `README.md`，保证文档与当前实现、目录约定、构建方式、关键组件使用方式保持一致。

## 列表与弹窗实现约定
- 页面数据流保持分层清晰：表格请求、弹窗开关、表单重置、提交、删除、状态切换分别使用独立方法，不要把业务逻辑直接塞进模板。
- 表格请求优先封装为独立 `requestXxxTable` 方法，并通过 `buildPageRequest` 统一处理分页参数。
- 批量删除和单项删除尽量复用同一个方法，统一兼容对象、对象数组、ID、ID 数组几种入参。
- 确认弹窗文案优先展示字段，`name`, `label`, `code`，推荐格式为“字段中文名：字段值”，例如“名称：xxx”“标题：xxx”；当前模型不存在`name`, `label`, `code` 字段，退回使用通用业务文案。
- 弹窗关闭时必须显式重置表单和校验状态，避免新增/编辑切换时残留旧数据。
- 编辑态数据回填、下拉选项预加载、提交成功后刷新表格，这三类行为必须在页面方法里显式处理，不要隐式耦合到基础组件内部。

## 组件扩展约定
- `ProTable` 的增强保持轻量，优先新增有限的配置项，不要把它做成复杂的业务 DSL。
- 适合沉淀到 `ProTable` 的能力仅限高复用、低业务耦合的表格展示能力，例如图片列、状态列、通用操作按钮。
- 不要把页面级业务流程、接口请求编排、权限判断分支的大量细节继续下沉到 `ProTable` 或 `FormDialog` 中。
- 若通用能力无法覆盖，优先保留页面侧扩展点，不要为了抽象而抽象。

## 协作约定
- 遇到跨组件、多页面、涉及基础组件接口调整、目录结构调整或生成文件策略调整这类较复杂改动时，正式修改前必须先列出清晰的修改步骤和影响范围，待确认后再实施。
- 若改动属于单文件小修、小范围样式调整、明确报错修复这类低风险任务，可直接修改，但仍需在开始前说明处理方向。

## 自动导入与类型文件约定
- `Element Plus` 运行时 API 和 `@element-plus/icons-vue` 图标优先通过 `unplugin-auto-import` 管理，不要在业务页面重复手写已有的 import。
- 类型导入仍然保持显式导入，例如 `FormRules`、`UploadUserFile`、`CheckboxValueType` 等类型，不通过自动导入隐式处理。
- 业务模型类型优先从 `src/rpc` 目录下的生成类型中引用，不要在页面、组件、hooks 中重复定义等价的业务类型。
- 仅当 `src/rpc` 中确实不存在所需结构，且该类型无法通过已有生成类型组合表达时，才允许在本地补充轻量类型；新增前优先评估是否应回到接口定义侧补齐生成类型。
- 自动生成文件统一放在 `types/generated` 目录，当前包括 `auto-imports.d.ts` 和 `eslint-auto-import.json`。
- `src/typings` 仅用于维护手写类型声明，不放自动生成文件。
- 若调整了自动导入配置，必须同步确认 `build/plugins.ts`、`.eslintrc.cjs`、`tsconfig.json` 的路径引用保持一致。

## 代码修改约束
- 后续凡是修改代码文件（包括但不限于 `.ts`、`.vue`、`.js`、`.scss`），在结束本次改动前必须执行 ESLint 检查。
- 当前项目默认 ESLint 命令为 `pnpm lint:eslint`。
- 若本次改动范围较小，也至少需要对受影响代码完成等效的 ESLint 校验，并处理报错后再结束任务。
- 若 ESLint 因仓库内历史遗留问题无法全量通过，必须在答复中明确说明本次执行范围、报错文件和阻塞原因，不能跳过不报。

## 注释规范
- 后续新增或修改代码时，必须补充中文注释，不能省略。
- 后续新增或修改方法时，必须补充中文方法注释，说明方法用途、关键参数、返回值或副作用。
- 后续新增或修改 `interface`、`type` 等类型定义时，必须补充中文类型注释；若类型字段语义不直观，字段也需要补充必要的中文注释。
- 后续新增或修改关键逻辑时，必须补充必要的中文行内注释，重点说明关键步骤、边界条件、异常分支和易误解的实现意图。
