# 推荐模块单机低配置实施计划

## 文档目的

本文档基于《`recommend-status-vs-gorse.md`》中的评估结论，给出当前项目推荐模块在单机、低配置服务器场景下的落地工作计划。

计划目标不是复刻 gorse 的完整平台，而是在当前工程基础上，做出一套：

- 能长期运行
- 资源消耗可控
- 推荐效果可持续优化
- 出问题能排查

的轻量推荐方案。

## 适用前提

本文默认以下部署约束成立：

- 单机部署
- 机器配置较低
- 推荐流量中低
- 允许离线任务在夜间运行
- 可以接受“部分用户 / 部分场景先优化，剩余流量回退当前在线链路”

## 总体策略

### 一：保留当前单体架构，不拆服务

当前阶段不拆：

- master
- worker
- server

继续沿用：

- 当前后端单体服务
- 当前 `base_job` 任务调度
- 当前推荐缓存协议
- 当前推荐版本表

这样做的原因是：

- 当前代码已经具备较完整推荐链路
- 单机环境下拆服务收益低，运维成本高
- 真实瓶颈不在“没有拆服务”，而在“没有最终推荐缓存”和“没有最小实验能力”

### 二：先做“小而稳”的推荐增强，不做“大而全”的平台建设

优先做：

- 活跃用户、首页场景的最终推荐离线缓存
- 最小可用灰度
- 推荐任务与效果可观测

暂缓做：

- 全量用户离线预生成
- 全场景离线最终推荐
- 通用 A-B 实验平台
- 多节点推荐集群
- 在线强依赖 LLM

## 目标与非目标

### 本阶段目标

1. 保证当前推荐业务闭环持续稳定运行
2. 在低资源前提下补齐最关键的平台短板
3. 让首页登录态推荐先具备“离线预生成 + 在线回退”
4. 让版本灰度从“配置字段”变成“真实能力”
5. 让后台能看见推荐任务状态和基本效果

### 本阶段非目标

1. 不追求全量用户、全场景最终推荐缓存
2. 不追求 gorse 式多进程部署形态
3. 不追求复杂实验平台
4. 不追求多模态内容理解
5. 不把 LLM 二次重排作为推荐核心路径

## 分阶段计划

## 阶段 P0：稳态治理

### 目标

先把当前推荐链路的运行稳定性、默认配置和排障信息补齐，避免在低配置机器上“功能越来越多，但越来越难维护”。

### 工作项

1. 梳理推荐任务的运行依赖、默认参数和执行顺序。
2. 明确推荐任务的夜间执行窗口，避免训练与白天流量竞争资源。
3. 统一推荐任务并发策略，单机环境下同一时刻只允许少量推荐任务运行。
4. 为关键任务补齐最近一次成功时间、耗时、输入规模、产出规模摘要。
5. 明确推荐链路的默认降级策略：
   - 缓存 miss 回退在线召回
   - 精排缓存 miss 回退规则排序
   - LLM 默认关闭

### 交付物

- 推荐任务运行基线说明
- 推荐任务执行顺序与调度建议
- 推荐链路默认降级清单

### 取舍说明

本阶段不新增重功能，只做稳定性与治理，因为单机低配环境最怕“链路复杂但没有观测”。

## 阶段 P1：首页登录态最终推荐缓存

### 目标

补齐当前项目与 gorse 的最大差距，但只做最小范围版本：

- 只做登录用户
- 只做首页场景
- 只做活跃用户
- 只做夜间批量生成

### 工作项

1. 新增“最终推荐结果”离线缓存子集合发布能力。
2. 新增首页登录态最终推荐生成任务。
3. 活跃用户筛选规则先从轻量方案开始，例如：
   - 最近 N 天有点击、收藏、加购、下单、支付任一行为
   - 或最近 N 天有推荐请求
4. 每个用户只生成较小 TopN，例如 50 或 100。
5. 在线首页推荐优先读取最终推荐缓存。
6. 缓存 miss 时，回退当前 `recommend_request.go` 在线动态链路。

### 交付物

- 最终推荐缓存发布能力
- 首页登录态最终推荐离线任务
- 首页推荐“缓存优先，在线回退”能力

### 取舍说明

本阶段明确不做：

- 匿名用户最终推荐缓存
- 购物车、订单详情、支付成功等长尾场景最终推荐缓存
- 全量用户全量场景统一预生成

这样做能用最小资源拿到最大收益，因为首页是最容易受益、最值得预生成的场景。

## 阶段 P2：最小可用灰度能力

### 目标

让当前 `grayRatio` 从“展示字段”变成“真实放量能力”，但只做单机低配可承受的最小方案。

### 工作项

1. 基于 `userId` 或匿名 `actorId` 实现稳定哈希分桶。
2. 单次只支持一个场景、两套版本对比：
   - 基线版本
   - 灰度版本
3. `grayRatio` 只控制流量占比，不扩展多层实验。
4. 将命中的版本信息写回请求上下文，便于后续评估。
5. 评估任务按版本或策略维度输出对比口径。

### 交付物

- 稳定分桶逻辑
- 真实灰度放量能力
- 按版本回流的评估口径

### 取舍说明

本阶段明确不做：

- 通用实验平台
- 多实验并行
- 嵌套实验
- 自动流量调度

单机环境先把“能稳定做单次灰度验证”做好，比“设计一套大实验系统”更重要。

## 阶段 P3：推荐后台与可观测能力

### 目标

让推荐系统从“能跑”进入“能运营、能排障”状态。

### 工作项

1. 在后台补推荐任务运行概览：
   - 最近执行时间
   - 最近执行结果
   - 最近耗时
   - 最近输入规模和输出规模
2. 补推荐缓存概览：
   - 各集合子集合数量
   - 最近更新时间
   - 命中摘要
3. 补推荐效果概览：
   - 请求数
   - 曝光数
   - 点击数
   - 支付数
   - CTR / CVR / NDCG
4. 在版本管理页补充更多排障字段入口，至少能定位：
   - 当前场景读取的是哪个版本
   - 是否命中最终推荐缓存
   - 是否回退到在线链路

### 交付物

- 推荐运行概览页或推荐工作台卡片
- 推荐缓存摘要
- 推荐效果摘要

### 取舍说明

本阶段优先做“汇总视图”和“最近状态”，不追求复杂图表和深度钻取，因为单机阶段核心目标是降低排障成本。

## 阶段 P4：内容能力与高阶能力谨慎增强

### 目标

在前几阶段稳定之后，再决定是否补更重的能力。

### 可选工作项

1. 升级内容相似表示方式
2. 接入外部 embedding
3. 引入更通用的外部推荐器接入层
4. 评估是否保留或扩展 LLM 二次重排

### 明确建议

以下能力默认延后，不进入近期主计划：

- 多模态相似
- 多进程推荐集群
- 通用推荐平台控制台
- 大规模全量用户离线预生成

## 推荐任务调度建议

单机低配置环境下，建议推荐任务全部放到低峰执行，并限制重任务并发。

建议顺序如下：

1. `RecommendGoodsStatDay`
2. `RecommendUserPreferenceRebuild`
3. `RecommendGoodsRelationRebuild`
4. `RecommendHotMaterialize`
5. `RecommendLatestMaterialize`
6. `RecommendSimilarItemMaterialize`
7. `RecommendSimilarUserMaterialize`
8. `RecommendCollaborativeFilteringMaterialize`
9. `RecommendContentBasedMaterialize`
10. `RecommendRankerMaterialize`
11. 首页登录态最终推荐缓存任务
12. `RecommendEvalReport`
13. `RecommendVersionPublish`

建议约束如下：

- 重训练任务串行执行
- 训练与评估不要同时跑
- 白天只跑轻量任务或不跑推荐重任务

## 默认配置建议

### 一：缓存

- 开发环境可接受内存缓存
- 单机生产环境仍建议保留小规格 Redis

原因：

- 最终推荐缓存、协同过滤缓存、精排缓存都更适合持久缓存
- 机器重启后不希望所有推荐缓存全部丢失

### 二：训练范围

- 默认只训练最近 30 到 90 天的数据
- 默认只处理启用版本
- 默认只处理活跃用户和上架商品

### 三：LLM

- 默认关闭在线 LLM 重排
- 若保留，仅用于人工实验或离线快照导入

## 验证策略

每个阶段都要明确“完成标准”，避免工作越做越散。

### P0 完成标准

- 推荐任务运行顺序明确
- 推荐任务默认参数明确
- 关键任务失败后能快速定位阶段

### P1 完成标准

- 首页登录态已支持最终推荐缓存
- 缓存 miss 不影响线上可用性
- 低资源机器上夜间任务可在可接受时间内完成

### P2 完成标准

- `grayRatio` 已真实影响流量
- 同一个用户命中版本稳定
- 评估结果能区分基线和灰度版本

### P3 完成标准

- 后台能看到推荐任务最近状态
- 后台能看到推荐效果摘要
- 出现效果异常时能快速定位到版本、缓存或任务问题

## 近期优先级建议

如果资源有限，建议严格按下面顺序推进：

1. `P0 稳态治理`
2. `P1 首页登录态最终推荐缓存`
3. `P2 最小可用灰度`
4. `P3 推荐后台与可观测`
5. `P4 高阶能力谨慎增强`

## 最终建议

单机低配置场景下，推荐模块最值得投入的方向不是“先做复杂平台”，而是：

- 用最小成本补齐最终推荐缓存
- 用最小成本补齐真实灰度
- 用最小成本补齐运行可见性

只要这三点落地，当前项目就能从“有推荐闭环”进一步升级为“有轻量平台能力的推荐系统”，且不会把低配置服务器拖垮。
