# 当前推荐逻辑和 参考实现 的差距说明

## 先说结论

当前项目里的推荐，已经不是“只有热门商品”那么简单了。

它已经有这些能力：

- 能区分匿名用户和登录用户
- 能记录推荐请求、曝光、点击、收藏、加购、下单、支付
- 能根据商品详情、购物车、订单这些场景给商品做推荐
- 能根据用户最近喜欢的商品、喜欢的分类、商品热度、商品新不新，综合排个序

但它整体上还是一套“业务规则型推荐”。
简单说，就是：

- 先按业务场景把候选商品找出来
- 再按一套提前写好的分数规则排顺序

参考实现 更像一套“通用推荐引擎平台”。
简单说，就是：

- 它不只是按规则排
- 它还会根据大量用户行为，自动训练模型
- 它能自己学“什么人喜欢什么东西”
- 它还带训练、缓存、监控、后台管理这些整套能力

所以两边最大的差别，不是“有没有推荐”，而是：

- 当前项目更懂电商业务场景
- 参考实现 更强在自动学习、自动调优、平台化能力

## 名词翻译

为了看表格时不绕，先把常见缩写翻成大白话：

| 缩写 | 通俗解释 |
| --- | --- |
| CF | 协同过滤。意思是“跟你口味相近的人喜欢什么，也推荐给你” |
| CTR | 点击率。比如展示 100 次，被点了 5 次，CTR 就是 5% |
| CVR | 转化率。比如点了 100 次，最后买了 3 次，CVR 就是 3% |
| FM | 一种排序模型。可以理解成“不是人工写死权重，而是让系统自己学怎么打分” |
| Embedding | 向量。可以理解成“把商品内容转成一串数字，方便算相似度” |
| Item to Item | 商品找相似商品。看起来像“买了这个的人，还可能喜欢那个” |
| User to User | 用户找相似用户。看起来像“跟你行为像的人，还喜欢什么” |
| Recall | 召回。先从大商品池里捞一批“可能相关”的商品 |
| Rank | 排序。把召回出来的商品，重新排先后顺序 |
| LLM | 大模型。比如用 AI 再帮你把候选商品重新排序 |

## 当前项目现在做到什么程度

用大白话说，当前项目的推荐链路是：

1. 用户来到某个场景，比如商品详情页、购物车页、订单页
2. 系统先按这个场景找一批候选商品
3. 再把用户最近喜欢过什么、买过什么、商品最近热不热、是不是刚上新，综合算分
4. 最后把推荐结果返回，并把这次请求记下来
5. 如果用户看了、点了、加购了、下单了，这些行为还会继续回流，给后面的推荐继续用

所以它已经是“有闭环”的，不是死数据。

## 参考实现 自己内部都用了哪些组件

如果把 参考实现 当成一整套系统来看，它不是一个单文件服务，而是几块东西一起干活：

| 组件 | 它是干什么的 | 通俗说法 |
| --- | --- | --- |
| Master | 负责训练模型、更新热门榜、更新相似商品、更新相似用户、管理配置 | 大脑 |
| Worker | 负责给每个用户批量算推荐结果，并把结果写进缓存 | 批量干活的工人 |
| Server | 负责对外提供接口，请求来了就把推荐结果吐出去 | 对外窗口 |
| Dashboard | 负责看任务状态、看配置、看系统运行情况 | 后台控制台 |
| Data Store | 存用户、商品、行为这些原始数据 | 原始仓库 |
| Cache Store | 存算好的推荐结果、相似商品、热门榜这些中间结果 | 快速取货区 |
| Meta Store | 存配置、模型信息、节点信息 | 记账本 |
| Blob Store | 存训练好的模型文件 | 模型仓库 |

### 用一句话理解这套分工

- Master 负责“学”
- Worker 负责“批量算”
- Server 负责“对外给结果”
- Dashboard 负责“让人看得见、管得住”

## 参考实现 的数据一般存在哪里

参考实现 里有两种“存东西”的地方，不是一锅炖。

### 1. 原始数据存储

这里放的是最基础的数据：

- 用户是谁
- 商品是什么
- 用户对商品做过什么事

代码里支持这些地方存原始数据：

| 存储类型 | 用来干什么 |
| --- | --- |
| MySQL | 存用户、商品、行为数据 |
| PostgreSQL | 存用户、商品、行为数据 |
| MongoDB | 存用户、商品、行为数据 |
| ClickHouse | 存用户、商品、行为数据，偏大数据量场景 |
| SQLite | 更适合本地测试或小规模跑通 |

### 2. 缓存存储

这里放的不是原始行为，而是“已经算好的结果”。

比如：

- 某个用户的推荐列表
- 某个商品的相似商品列表
- 某个用户的相似用户列表
- 热门榜、最新榜
- 推荐结果更新时间
- 各种摘要信息和评估指标

代码里支持这些地方做缓存：

| 存储类型 | 用来干什么 |
| --- | --- |
| Redis | 最常见，拿结果快 |
| MySQL | 也能当缓存存储用 |
| PostgreSQL | 也能当缓存存储用 |
| MongoDB | 也能当缓存存储用 |
| SQLite | 本地测试可用 |

### 3. 模型文件单独存

参考实现 训练完的模型，不是直接塞进推荐结果缓存里。
它会把模型文件单独存起来，方便 Worker 拉下来使用。

你可以把它理解成：

- 数据库存事实数据
- 缓存库存算好的结果
- 模型仓库存“训练成果”

## 参考实现 缓存里到底保存了什么

这个问题最容易问得很实际，下面直接用人话列出来。

| 缓存内容 | 是什么 |
| --- | --- |
| `non-personalized` | 非个性化榜单，比如热门、最新 |
| `item-to-item` | 某个商品对应的一组相似商品 |
| `user-to-user` | 某个用户对应的一组相似用户 |
| `collaborative-filtering` | 协同过滤算出来的用户候选结果 |
| `recommend` | 给某个用户最终算好的推荐列表 |
| `*_digest` | 当前这份结果对应的版本指纹，可以理解成“这份缓存是不是过期了” |
| `*_update_time` | 这份缓存最后一次更新时间 |
| `global_meta` | 全局统计信息，比如用户数、商品数、训练时间、评估指标 |
| `item_categories` | 商品分类信息，方便筛分类推荐 |

所以 参考实现 的缓存不是只放“最后推荐结果”，而是把推荐链路里很多中间结果都缓存了。
这样做的好处是：

- 下次请求更快
- 不用每次重新现算
- 训练和对外服务可以分开

## 参考实现 自动学习，主要靠哪些组件

如果用大白话说，参考实现 的“自动学习”不是靠一个东西，而是好几个模块配合。

### 第一类：学“哪些商品彼此像”

这类模块会做：

- 商品找相似商品
- 看标签像不像
- 看用户群体是不是经常一起喜欢
- 看商品向量是不是接近

你可以把它理解成：

- “看过这个的人，还可能会对哪个感兴趣”

### 第二类：学“哪些用户彼此像”

这类模块会做：

- 找行为相似的用户
- 找标签相似的用户
- 找买过相似商品的用户

你可以把它理解成：

- “和你差不多的人，最近喜欢什么”

### 第三类：学“用户大概率会喜欢什么”

这部分就是 `CF`，也就是上面说的协同过滤。

通俗说法就是：

- 不只是看你自己做过什么
- 还会看整站很多人的行为规律
- 最后学出一套“谁更可能喜欢什么商品”的模型

这块在 参考实现 里会训练专门的模型，然后产出“用户向量”和“商品向量”，后面 Worker 会拿这些结果去批量算推荐。

### 第四类：学“这个推荐被点开的概率高不高”

这部分就是 `CTR` 模型。

通俗说法就是：

- 候选商品已经有了
- 系统还会再判断一下：这些候选里，哪个更容易被点、哪个更值得排前面

所以它不是只会“找候选”，还会“帮你重新排顺序”。

### 第五类：自动挑更好的参数

参考实现 不只是训练一次就不管了。
它还会定时试不同参数，比较哪套更好。

通俗说法就是：

- 同样是推荐模型
- 它会试试 A 方案、B 方案、C 方案
- 看谁的效果更好
- 再把更好的那套留下来

这一步就是当前项目和 参考实现 差距比较大的地方之一。

## 用流程图思维理解 参考实现 的自动学习

可以把 参考实现 的自动学习想成下面这几步：

1. 先把用户、商品、行为数据从数据库里拉出来
2. 再生成热门榜、相似商品、相似用户这些中间结果
3. 再训练“用户可能喜欢什么”的模型
4. 再训练“哪个结果更容易被点击”的排序模型
5. 再把训练好的模型文件存起来
6. Worker 再用这些模型，批量给每个用户算推荐结果
7. 最后把结果写到缓存里，给 Server 对外返回

所以它的自动学习不是“来一个请求现学一次”，而是：

- 后台先学
- 学完批量算
- 线上直接拿结果

## 如果只看“参考实现 靠什么实现自动学习”

可以简单理解成下面这张表：

| 能力 | 主要靠什么 |
| --- | --- |
| 热门榜、最新榜 | 非个性化推荐模块 |
| 商品相似商品 | `item-to-item` 模块 |
| 用户相似用户 | `user-to-user` 模块 |
| 用户兴趣学习 | `CF` 模型 |
| 点击概率排序 | `CTR` 模型 |
| 模型参数自动优化 | 自动调参模块 |
| 批量产出推荐结果 | Worker + Pipeline |
| 对外快速返回 | Server + Cache |

## 用一句人话概括每个关键差距

### 1. 当前项目更像“老师傅按经验推荐”

现在这套推荐很像一个懂业务的运营同事在做判断：

- 你刚看了什么商品
- 你购物车里有什么
- 你最近喜欢什么分类
- 哪些商品最近比较热
- 哪些商品最近别老推了

然后根据这些经验，给每个商品加减分。

这种方式的优点是：

- 容易理解
- 容易改
- 很符合电商业务

缺点是：

- 学习能力有限
- 用户一多、商品一多后，人工规则会越来越重

### 2. 参考实现 更像“会自己总结规律的系统”

参考实现 不只是写规则，它会看全站用户行为，自己学规律。

比如它会慢慢学出来：

- 买 A 的人，后来大概率也会买 B
- 跟你行为相似的人，最近都在看 C
- 某一类人群，对某类商品更容易点开
- 某种组合下，推荐 D 比推荐 E 更容易成交

这就是为什么 参考实现 更像推荐引擎，而不是推荐业务代码。

### 3. 当前项目缺的最大块，不是埋点，而是“自动学习”

这一点最容易看错。

当前项目不是没有数据，也不是没有闭环。
它已经能记：

- 请求
- 曝光
- 点击
- 收藏
- 加购
- 下单
- 支付

所以“数据采集”这块已经不差了。

真正差的是：

- 这些数据现在主要拿来做统计、加权、重建
- 还没有充分拿来训练“会自己变聪明”的推荐模型

## 如果按优先级补差距，建议顺序

如果后面想逐步靠近 参考实现，不建议一下子全重做。
更合适的顺序是下面这样：

| 优先级 | 建议补什么 | 通俗解释 |
| --- | --- | --- |
| 第一优先级 | 补“相似用户”和“协同过滤”能力 | 不只是看你自己喜欢什么，还看和你像的人喜欢什么 |
| 第二优先级 | 补模型排序 | 不再完全靠人工写死权重，让系统自己学怎么排更好 |
| 第三优先级 | 补内容相似能力 | 不只看行为统计，还看商品内容本身是不是像 |
| 第四优先级 | 补离线缓存推荐 | 先把很多用户的推荐算好，线上拿结果更快 |
| 第五优先级 | 补版本切换和实验能力 | 能比较两套推荐谁更好，而不是只能人工感觉 |
| 第六优先级 | 补推荐后台 | 能看到任务状态、效果走势、版本情况，方便运营和排查 |

## 这份差距怎么映射到当前重构计划

如果不把差距映射到具体阶段，这份文档很容易变成“看上去很懂，但落不下去”的说明文。

当前仓库已经单独维护了 `recommend-rebuild-plan.md`，两份文档的关系建议固定为：

| 本文里的差距 | 在重构计划里落到哪一阶段 | 为什么放这里 |
| --- | --- | --- |
| 在线推荐逻辑太重 | 阶段 1、阶段 6 | 先抽边界，再切统一引擎，不能一步跳过去 |
| 行为事实和聚合耦合 | 阶段 2 | 训练、缓存、评估都依赖稳定事实层 |
| 没有缓存优先和预计算结果 | 阶段 3 | 这是从“在线现算”转向“缓存优先”的关键一步 |
| 没有相似用户、协同过滤、内容相似 | 阶段 4、阶段 5 | 先补召回框架，再补训练产物 |
| 没有模型排序 | 阶段 5、阶段 7 | 先有训练和特征，再有稳定精排和重排 |
| 没有自动调参和版本闭环 | 阶段 8 | 这一步依赖评估口径、训练结果、发布协议同时存在 |
| 没有推荐后台和监控 | 阶段 9 | 前面没有稳定指标和状态，后台也只是空壳 |

换句话说，这份差距说明文档回答的是“为什么要改”，`recommend-rebuild-plan.md` 回答的是“按什么顺序改”。

## 当前阶段最该补什么

如果结合仓库当前状态看，最现实的重点不是直接上 CF 或 LLM，而是先把下面两件事做扎实：

1. 继续完成阶段 2，把事实层和投影层边界彻底收口。
2. 紧接着启动阶段 3，把热门榜、最新榜、相似商品这些最稳定的结果先物化到缓存。

原因很直接：

- 没有稳定事实层，后面的训练输入就不可信。
- 没有缓存物化层，即使补了更多召回和排序能力，线上仍然会被“每次请求现算”拖住。
- 没有版本化缓存，后面的灰度、回滚、实验都很难做。

如果站在业务角度看：

- 当前项目这套推荐，已经能用，而且很贴商城场景
- 它适合“先把业务跑起来，推荐结果可解释、可控”
- 参考实现 更适合“后面要把推荐做深、做成平台、做自动学习”

所以这不是“当前项目没做推荐”，而是：

- 当前项目已经有一套实用推荐
- 但离真正的推荐引擎平台，还有一段距离

## 一句话版总结

当前项目更像“懂电商业务的规则推荐”，参考实现 更像“会自己学习、自己训练、自己调优的推荐平台”。
