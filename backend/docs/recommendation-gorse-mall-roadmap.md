# 商城专用 Gorse 化推荐服务路线图

本文用于统一推荐改造方向：目标不是把 Gorse 原样搬进来做“通用推荐平台”，而是吸收 Gorse 的能力模型，落成一套只服务当前商城业务的推荐服务。

结论时间：2026-04-12

## 最终目标

最终形态应是：

- 只服务商城业务，不做多租户、跨行业、任意资源类型的通用推荐引擎。
- 保留当前商城的业务上下文能力，例如首页、购物车、订单详情、支付成功等场景语义。
- 继承 Gorse 的核心能力，包括：
  - 用户 / 商品 / 反馈的标准化建模
  - 多路召回
  - 排序 / 重排
  - 离线训练与重建
  - 在线评估与离线评估
  - 任务编排与可观测能力
- 推荐服务的数据事实仍然以当前商城业务库为主，不额外复制整套业务主数据作为新的“真相源”。

## 三层边界

后续改造必须严格分成 3 层，不要混写。

### 1. 商城业务事实层

这一层负责商城真实业务，不负责推荐算法。

包含：

- 用户
- 商品
- 购物车
- 收藏
- 订单
- 支付
- 商品上下架、库存、价格、会员价、类目、活动

当前主要对应：

- `base_user`
- `goods_info`
- `user_cart`
- `user_collect`
- `order_goods`
- 订单和支付相关表

这一层的职责是：

- 提供真实业务事实
- 作为推荐数据源
- 不承载推荐模型和推荐策略

### 2. 商城推荐域层

这一层是整个改造的核心。

它不属于通用 Gorse，也不属于商城业务基础 CRUD，而是你的“商城专用推荐语义层”。

包含：

- 场景定义
  - 首页
  - 购物车
  - 订单详情
  - 支付成功
- 反馈语义定义
  - 曝光
  - 点击
  - 浏览
  - 收藏
  - 加购
  - 下单
  - 支付
- 推荐上下文
  - `scene`
  - `request_id`
  - `position`
  - 订单上下文
  - 购物车上下文
- 业务过滤与重排
  - 库存过滤
  - 上下架过滤
  - 重复购买惩罚
  - 类目打散
  - 会员商品或活动商品过滤
- explain、归因、评估口径

这一层要解决的是：

- 把商城业务事实转成推荐语义
- 把 Gorse 能力变成商城可用能力
- 承接业务策略，不污染 Gorse 内核

### 3. Gorse 内核层

这一层只负责推荐引擎能力，不关心商城业务细节。

建议下沉到这一层的能力：

- retrieval
- ranking
- reranking
- user/item/feedback 抽象
- 模型训练
- 模型重建
- 离线推荐生成
- 评估任务
- 任务编排

这一层不应该出现的内容：

- 购物车规则
- 订单状态规则
- 支付业务语义
- 库存和价格判定
- 活动规则
- 商城页面场景常量

## 边界原则

一句话原则：

- `业务事实留在商城业务层`
- `业务语义收敛到商城推荐域层`
- `算法能力沉到 Gorse 内核层`

因此，不建议：

- 把订单、购物车、支付逻辑塞进 Gorse core
- 直接修改 Gorse 内核去理解商城场景
- 让 Gorse 直接依赖商城业务表结构

建议：

- 让商城推荐域层去读业务事实
- 让商城推荐域层把数据适配成内核可用语义
- 让 Gorse 内核只处理推荐计算本身

## 哪些能力应该“复制到系统内”

你说的“把能力复制到系统内”，正确拆法是：

### 应复制进系统内的

- Gorse 的召回框架
- Gorse 的排序 / 重排能力
- Gorse 的训练任务
- Gorse 的重建任务
- Gorse 的评估任务
- Gorse 的模型生命周期管理

### 不应直接复制进系统内的

- Gorse 的通用产品边界
- Gorse 面向任意资源集合的抽象设计
- Gorse 的通用 dashboard 假设
- Gorse 对非商城领域的兼容设计

### 应由商城推荐域自己负责的

- 场景定义
- 反馈映射
- 过滤规则
- 归因体系
- explain 口径
- 策略发布和灰度规则

## 为什么不能把业务直接下沉到 Gorse

如果把业务直接下沉到 Gorse，会出现 4 个问题。

### 1. 业务和引擎耦合过深

以后你改一个商城规则，就会改到推荐内核。

### 2. Gorse 升级成本会急剧上升

一旦深 fork，后续再吸收上游能力会非常痛苦。

### 3. 推荐引擎会失去稳定边界

引擎层应该稳定，商城规则应该高频变化。

### 4. 业务语义会被抽象污染

商城里的“支付成功页顺手带一件”和通用推荐语义不是一回事，硬塞进通用内核会让结构越来越乱。

## 不建议的做法

不建议直接按下面方式推进：

- 先深度改 Gorse 源码，再把商城业务硬塞进去。
- 先把 Gorse 所有通用表、缓存表、文档表一次性补齐。
- 先做独立推荐平台，再回头接商城业务。

原因很简单：

- 你当前的真实优势在于已有完整商城链路和业务上下文，不在于重新做一个通用推荐底座。
- 如果先按通用平台设计，表会变多、抽象会变厚、业务语义会被削弱。
- 后续升级 Gorse 或继续吸收其能力时，硬 fork 的维护成本会很高。

## 建议原则

### 1. 业务表继续做事实源

下面这些表继续做推荐数据的原始事实来源：

- `base_user`
- `goods_info`
- `user_cart`
- `user_collect`
- `order_goods`
- `recommend_request`
- `recommend_exposure`
- `recommend_goods_action`

也就是说：

- 用户事实在业务表
- 商品事实在业务表
- 行为事实在推荐原始表和订单表

推荐系统不要再复制出一套新的业务主表去替代它们。

### 2. 推荐服务只新增“推荐域自己的表”

推荐服务应该拥有的，不是通用业务主表，而是推荐域自己的三类表：

- 跟踪归因表
- 模型与策略表
- 评估与任务结果表

这是你和 Gorse 的真正结合点。

### 3. 先做商城专用适配层，再吸收 Gorse 能力

正确顺序应该是：

1. 先把当前业务数据整理成商城推荐内部统一语义。
2. 再把 Gorse 的 retrieval / ranking / evaluation / training 能力一层层接进来。
3. 最后再决定是否拆成独立推荐服务进程。

而不是反过来先做通用引擎。

## 推荐目标架构

建议把后续推荐体系拆成 5 层。

### 1. 业务事实层

职责：

- 提供用户、商品、订单、购物车、收藏、支付等原始事实
- 不承载推荐算法状态

对应当前表：

- `base_user`
- `goods_info`
- `user_cart`
- `user_collect`
- `order_goods`

### 2. 推荐事件与归因层

职责：

- 保存推荐请求
- 保存逐商品曝光与逐商品行为
- 保存匿名到登录主体的绑定事实

这一层是商城专用推荐最关键的一层，因为 Gorse 原生 `feedback` 不关心你商城里的“第几个坑位、哪个场景、哪次请求”。

### 3. 推荐标准化适配层

职责：

- 把商城业务事实转换成推荐内部统一语义
- 对外暴露内部统一的：
  - `user`
  - `item`
  - `feedback`
  - `context`

这里建议优先做“代码适配层”或“视图/ETL”，而不是一开始就建三张永久复制表。

例如：

- `base_user` -> 推荐用户画像
- `goods_info` -> 推荐商品画像
- `recommend_goods_action` + `order_goods` -> 正反馈 / 已读反馈

### 4. 推荐计算层

这层就是你真正要继承 Gorse 能力的地方。

建议逐步继承：

- latest / hot
- item-to-item
- user-to-user
- collaborative filtering
- 内容特征召回
- ranker / reranker
- model selection
- offline recommendation generation

但这些能力都必须以商城语义落地，例如：

- 首页猜你喜欢
- 购物车凑单
- 订单详情搭配购
- 支付成功顺手带一件

而不是面向任意资源集合。

### 5. 评估与运维层

职责：

- 评估任务
- 模型版本
- 策略发布
- 看板与 explain

这是 Gorse 价值很高但你当前链路仍然缺的一层。

## 哪些 Gorse 能力要继承

建议明确继承以下能力，而不是继承整套通用抽象。

### 必继承

- 用户 / 商品 / 反馈的统一语义
- 多路召回机制
- 排序与重排
- 训练任务与离线重建
- 评估体系
- 任务调度与状态管理
- explain 与效果分析

### 选择性继承

- embedding / 多模态内容召回
- LLM reranker
- 离线全量推荐缓存
- dashboard 交互能力

### 不建议优先继承

- 通用文档存储抽象
- 任意集合资源推荐
- 面向多行业的通用 API 设计
- 通用 tenant / namespace 抽象

## 表结构应该怎么收敛

如果目标是“商城专用 Gorse 化”，推荐域自己的 MySQL 表建议控制在两批。

## 第一批：必须补

这些表是必须的，因为它们直接解决当前商城推荐链路缺口。

### 1. `recommend_request_item`

作用：

- 保存一次推荐请求返回的逐商品结果、位置和分数

为什么必须：

- 没有这张表，后续 CTR / CVR / 位置分析、召回源分析和 explain 回放都会继续依赖 JSON 字段。

### 2. `recommend_exposure_item`

作用：

- 把曝光从批量 JSON 归一成逐商品行

为什么必须：

- 没有这张表，很难稳定做曝光到点击、曝光到下单的 MySQL 级归因。

### 3. `recommend_actor_bind_log`

作用：

- 保存匿名主体绑定登录主体的事实

为什么必须：

- 没有这张表，后续重放、补偿、审计都缺锚点。

### 4. `recommend_model_version`

作用：

- 保存商城专用推荐策略或模型版本

为什么必须：

- 你未来一旦开始继承 Gorse 的训练和排序能力，就必须知道线上当前是哪套策略。

### 5. `recommend_eval_report`

作用：

- 保存评估结果

为什么必须：

- 没有评估结果表，Gorse 的评估能力继承进来也落不到库里。

## 第二批：优先用适配层，不急着物理建表

下面这些不是当前必须新增的实体表。

### 1. `gorse_users`

不建议第一阶段就落物理表。

建议做法：

- 先通过代码适配或 SQL 视图，把 `base_user` 转成推荐内部 `user` 语义。

### 2. `gorse_items`

不建议第一阶段就落物理表。

建议做法：

- 先通过代码适配或 SQL 视图，把 `goods_info` 转成推荐内部 `item` 语义。

### 3. `gorse_feedback`

不建议第一阶段就落物理表。

建议做法：

- 先通过代码适配、视图或离线导出，把：
  - `recommend_goods_action`
  - `recommend_exposure`
  - `order_goods`
  - `user_collect`
  - `user_cart`
  统一转换成推荐内部 `feedback`。

原因：

- 这三张表本质上是“标准化接口层”，不是新的业务事实源。
- 第一阶段物理落表，只会增加双写和一致性维护成本。

## 第三批：现在可以不做

下面这些是 Gorse 在通用场景里可能需要的能力，但对你当前商城专用目标并不紧急。

- `values`
- `message`
- `documents`
- `time_series_points`
- 通用文档型内容库
- 通用多租户管理表

除非你明确决定把 MySQL 也当推荐缓存存储，否则这一批先不要做。

## 推荐落地顺序

## 阶段一：稳住当前商城推荐链路

目标：

- 让现有推荐真正可分析、可重放、可评估

动作：

1. 新增 `recommend_request_item`
2. 新增 `recommend_exposure_item`
3. 新增 `recommend_actor_bind_log`
4. 把下单、支付行为改成以后端订单事实自动回写

## 阶段二：做商城专用推荐语义适配层

目标：

- 不建新的业务主表前提下，把当前业务事实整理成推荐内部统一语义

动作：

1. 定义内部 `User / Item / Feedback / Context` 结构
2. 编写 `base_user -> User`
3. 编写 `goods_info -> Item`
4. 编写 `goods_action / exposure / order -> Feedback`
5. 编写场景级 `ContextAdapter`

这里优先做代码适配，不先追求物理标准表，也不复制整套业务主表。

## 阶段三：继承 Gorse 的算法和任务能力

目标：

- 把 Gorse 的价值能力逐项嫁接到商城推荐

动作：

1. 接入 item-to-item
2. 接入 user-to-user
3. 接入 collaborative filtering
4. 接入训练任务
5. 接入重建任务
6. 接入评估任务
7. 新增 `recommend_model_version`
8. 新增 `recommend_eval_report`

## 阶段四：按三层边界拆服务

目标：

- 从当前业务后端中抽出推荐核心服务

建议拆分边界：

- 商城业务服务继续维护用户、商品、订单事实
- 商城推荐域服务负责：
  - 推荐请求
  - 推荐事件
  - 业务语义适配
  - 场景策略
  - explain 与归因
- Gorse 化推荐引擎负责：
  - 召回排序
  - 模型训练
  - 评估任务
  - 重建任务

这时再考虑是否要更深地复用或改造 Gorse 的运行时架构。

## 你现在最应该做什么

如果按最终目标倒推，当前最正确的动作不是“把 Gorse 所有表先补齐”，而是下面这 6 步：

1. 明确三层边界：业务事实层、商城推荐域层、Gorse 内核层。
2. 明确推荐服务只做商城业务，不做通用推荐平台。
3. 保留业务表做事实源，不复制一套新的业务主表。
4. 先补归因表和评估表，不急着补通用兼容表。
5. 先做商城专用 `user/item/feedback/context` 语义适配层。
6. 再逐步吸收 Gorse 的召回、排序、训练和评估能力。

## 一句话结论

你要做的不是“把 Gorse 改成商城系统”，而是“以三层边界为约束，把当前商城推荐系统演进成一个商城专用、能力上 Gorse 化的推荐服务”。
