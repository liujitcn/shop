# 模块设计方案

## 1. 目标

本仓库以 `base`、`system` 作为可独立运行的基础能力，以 `shop` 作为第一个可插拔、可删除的业务模块。后续新增业务应复用这套模块边界，不能把业务概念继续放进基础层。

模块是一组接口明确、实现可独立演进的业务能力。调用方只需理解其必要规则；删除模块时，复杂度应随模块目录一同消失。`shop` 是当前参考实现，不是基础脚手架的前置依赖。

本方案约束：

1. 基础能力不感知具体业务模块。
2. 业务模块通过少量显式接缝接入宿主进程。
3. 新增、删除或替换业务模块时，影响集中在模块目录、组合根和部署数据。

## 2. 当前结构

```text
frontend/admin, frontend/app
        │
        │  使用生成的 TypeScript RPC 类型
        ▼
backend/api/proto/{base,common,system,<module>}
        │
        ▼
backend/service/{base,system,<module>}
        │
        ▼
backend/server/{base,system,<module>}
        │
        ▼
backend/internal/cmd/server
        ├── 显式组合 Wire ProviderSet
        ├── 注册 HTTP / gRPC / MCP 服务模块
        └── 汇总任务、事件订阅、SSE 与 AI 扩展

backend/pkg
        └── 模块无关的运行时、事件、SSE、任务和基础设施能力
```

| 区域 | 职责 | 不应承担的职责 |
| --- | --- | --- |
| `pkg` | 通用基础设施、模块无关的稳定接口与运行时能力 | 商品、订单、推荐、评价等业务语义 |
| `service/base` | 认证、文件、通用 AI、通用 SSE、配置等基础业务 | 依赖任何业务模块或业务协议 |
| `service/system` | 用户、租户、权限、代码生成等系统管理能力 | 调用某个业务模块完成系统主流程 |
| `service/<module>` | 模块领域规则、常量、业务事件和第三方适配 | 向基础层泄漏模块专属类型 |
| `server/<module>` | 模块 HTTP、gRPC、MCP 和任务适配 | 承载领域规则或跨模块编排 |
| `internal/cmd/server` | 唯一组合根，声明当前启用的模块 | 隐藏注册、业务实现或运行时扫描 |
| `api/proto/<module>` | 模块对前后端的契约 | 模块私有以外的伪通用类型 |
| `frontend/*/src/{api,views}/<module>` | 模块前端请求适配和页面实现 | 把模块状态、类型或页面逻辑留在基础目录 |

## 3. 依赖边界

```text
pkg / service/base / service/system
  └── 不得 import service/<module> 或 api/gen/go/<module>

service/<module>
  └── 可以依赖 pkg、base、system 及其稳定协议

server/<module>
  └── 可以依赖该模块 service 和协议，并实现宿主接缝

internal/cmd/server
  └── 唯一允许同时引用基础层和全部已启用业务模块的位置
```

`common` 协议只保留真正跨模块的稳定类型。模块的枚举、SSE payload、常量、工作台刷新原因或第三方集成类型，必须留在该模块协议或该模块源码中。

业务模块之间默认不直接互相 import。一个模块需要感知另一个模块的变化时，优先由基础层定义不含业务语义的事件或高层接口，再由组合根显式装配业务侧适配器。只有稳定且不可替代的领域关系才允许新增跨模块契约，并应记录其所有权。

禁止用 `init()`、运行时扫描、反射注册或 build tag 隐藏模块启用状态。模块是否启用必须能从组合根的 import、Wire ProviderSet 和 `newModules` 参数直接看出。

## 4. 模块接缝

接缝是基础层允许业务模块接入的位置。只有确实存在可变行为时才实现接缝；不要为了统一形式增加空 Provider、Registry 或单行转发接口。

| 场景 | 基础层接口 | 模块负责的实现 | 组合方式 |
| --- | --- | --- | --- |
| 用户数据变化 | `pkg/event.UserEvents`、`UserSubscriber` | 将用户变化转换为模块异步工作 | 在 `newModules` 中显式订阅 |
| SSE 推送 | `pkg/sse.Stream`、`Registry`、`Publisher` | 流标识、事件标识、payload 和订阅解析 | 在模块 Provider 中注册 |
| 定时任务 | `server.TaskContributor`、`pkg/job.Task` | 稳定任务名和执行器 | 在 `newModules` 中汇总注册 |
| AI 固定流程 | `service/base/agent/ai.FixedFlowProvider` | 流程、入口动作、回复和快捷入口 | 在模块 Provider 中 `RegisterFixedFlow` |
| 管理端代码生成 | `service/system/admin/codegen.ProtoTarget` | 协议、服务和前端目录元数据 | 在模块 Provider 中显式注册 |

SSE 流标识、任务名、AI 流程标识和代码生成目标重复时必须启动或生成失败，不能以后注册覆盖先注册。数据库中启用但代码未贡献的定时任务也必须阻止启动，避免配置错误被静默忽略。

### 4.1 用户事件

基础用户、租户和认证流程只发布 `UserChanged`、`UsersDeleted` 这类通用事实，不知道消费者是推荐、积分还是 CRM。模块订阅者只能做不影响用户主流程的轻量处理；需要可靠投递时使用 outbox 等业务方案，而不是扩大进程内事件总线。

### 4.2 SSE

`pkg/sse` 只解决连接、鉴权后的流解析与 JSON 发布。流名、事件名、payload 和页面刷新语义全部由业务模块拥有。例如商城工作台使用 `shop.admin.workspace` 流和商城私有刷新枚举，这些定义不能回流到 `base` 或 `common`。

### 4.3 定时任务与 AI

模块通过 `TaskContributor` 返回 `[]job.Task`。任务名是数据库 `base_job.invoke_target` 的稳定契约，改名必须同步迁移数据。

基础 AI 提供通用对话和固定流程注册表。业务模块只在需要固定流程、入口动作或快捷入口时实现 `FixedFlowProvider`，并声明唯一流程标识；没有模块注册时，基础 AI 仍必须可用。

## 5. 新增业务模块

以下以新增 `<module>` 为例。按业务是否面向管理端、商城端或两端选择目录，不需要的端不创建空目录。

1. **明确接口与数据所有权**：先定义模块解决的问题、拥有的表、对外协议、依赖的基础能力，以及是否需要事件、SSE、任务、AI 或代码生成。
2. **定义协议**：在 `backend/api/proto/<module>/admin/v1` 或 `backend/api/proto/<module>/app/v1` 定义接口；只有跨端共享类型才进入 `common/v1`。表字段映射保持数据库字段顺序和连续编号。
3. **实现后端**：在 `backend/service/<module>` 放领域规则、常量、私有事件和第三方适配；在 `backend/server/<module>/admin`、`backend/server/<module>/app` 实现 `server.Module`。只在实际需要时提供任务集合或扩展 Provider。
4. **显式组合**：新建 `backend/internal/cmd/server/wire_<module>.go` 集中 ProviderSet、传输服务、任务和订阅者；只在 `wire.go`、`modules.go` 和必要时的 `main.go` 启用它。生成的 Wire 文件只由 `make wire` 更新。
5. **实现前端**：先生成 RPC 类型。管理端请求放 `frontend/admin/src/api/<module>`，页面和私有工具放 `frontend/admin/src/views/<module>`。商城端页面按既有主包、分包和 `pages.json` 组织。页面、组件和 store 优先使用 `src/rpc` 生成类型。
6. **同步数据库**：新增表、菜单、接口权限、Casbin 规则、字典、配置或任务时，在 `sql` 提供初始化或迁移脚本。`pkg/gen` 按实际数据库表生成，不手工编辑。
7. **生成与验证**：协议、实现、前端和 SQL 按上述顺序同步，再执行对应生成和检查命令。

## 6. 生成与验证

接口或模块结构变更后：

```bash
cd backend
make api openapi ts wire
go build ./...
go test ./...
go vet ./...
make -C .. check-boundary
```

变更数据库表时，连接目标数据库后再执行：

```bash
cd backend
make gorm-gen
go test ./...
```

管理端代码变更执行 `pnpm type:check` 与 `pnpm lint:oxlint`；商城端代码变更执行 `pnpm lint` 与 `pnpm tsc`。只修改文档时，至少检查 Markdown diff 和 `git diff --check`。

## 7. 删除模块与脚手架演进

符合本方案的业务模块可以从基础脚手架移除。应在独立分支完成：

1. 删除模块的 `service`、`server`、`api/proto`、生成协议、前端页面与请求适配目录。
2. 删除 `wire_<module>.go`，并从 `wire.go`、`modules.go`、`main.go` 移除组合项。
3. 移除构建脚本中只为该模块存在的目录假设。
4. 清理菜单、接口权限、Casbin 规则、配置和启用中的定时任务；业务表数据保留、归档或销毁单独决策。
5. 执行 `go mod tidy`，重新生成协议、Wire、依赖和数据库模型，再完成全量构建、测试和静态检查。

删除不应要求修改 `pkg`、`service/base` 或 `service/system` 的业务代码。若仍需清理业务类型，应先补齐边界，不保留临时兼容层。

## 8. 开发评审清单

- [ ] 业务概念、常量、枚举和 payload 是否只存在于所属模块？
- [ ] `pkg`、`service/base`、`service/system` 是否没有导入该模块服务或协议？
- [ ] 事件、SSE、任务、AI 或代码生成接入是否确实需要，并通过显式组合根注册？
- [ ] 协议、后端实现、前端调用和 SQL 初始化数据是否已同步？
- [ ] 生成物是否全部由项目命令生成？
- [ ] 删除模块时，是否只需删除模块目录、组合根引用和部署数据？
- [ ] 是否完成了与改动范围匹配的构建、测试、静态检查和文档更新？
