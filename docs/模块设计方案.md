# 模块设计方案

## 1. 目标

本仓库以 `base`、`system` 作为稳定基础能力，以管理后台作为当前唯一前端入口。业务能力应保持在独立模块中，通过组合根显式装配，避免业务概念泄漏到基础层。

模块是一组接口明确、实现可独立演进的能力。调用方只依赖公开契约；删除模块时，复杂度应随模块目录、协议和部署数据一同消失。

本方案约束：

1. 基础能力不感知具体业务实现。
2. 业务模块通过少量显式接缝接入宿主进程。
3. 新增、删除或替换业务模块时，影响集中在模块目录、组合根和部署数据。

## 2. 当前结构

```text
frontend/admin
        │
        │  使用生成的 TypeScript RPC 类型
        ▼
backend/api/proto/{base,common,system}
        │
        ▼
backend/service/{base,system}
        │
        ▼
backend/server/{base,system}
        │
        ▼
backend/internal/cmd/server
        ├── 显式组合 Wire ProviderSet
        ├── 注册 HTTP / gRPC / MCP 服务
        └── 汇总任务、事件订阅、SSE 与 AI 扩展

backend/pkg
        └── 模块无关的运行时、事件、SSE、任务和基础设施能力
```

| 区域 | 职责 | 不应承担的职责 |
| --- | --- | --- |
| `pkg` | 通用基础设施、稳定接口与运行时能力 | 具体业务语义 |
| `service/base` | 认证、文件、通用 AI、SSE、配置等基础业务 | 依赖某个业务模块 |
| `service/system` | 用户、租户、权限、代码生成等系统管理能力 | 调用业务模块完成系统主流程 |
| `server` | HTTP、gRPC、MCP 和任务适配 | 承载领域规则 |
| `internal/cmd/server` | 唯一组合根，声明当前启用的模块 | 隐藏注册或运行时扫描 |
| `api/proto` | 对外接口契约 | 伪通用的模块私有类型 |
| `frontend/admin/src` | 管理后台页面、请求适配和生成类型 | 把业务状态留在基础目录 |

## 3. 依赖边界

```text
pkg / service/base / service/system
  └── 不得 import 某个业务模块的服务或协议

业务模块
  └── 可以依赖 pkg、base、system 及其稳定协议

internal/cmd/server
  └── 唯一允许同时引用基础层和全部已启用业务模块的位置
```

`common` 协议只保留真正跨模块的稳定类型。模块枚举、SSE payload、常量和第三方集成类型必须留在所属模块。

业务模块之间默认不直接互相 import。需要感知其他模块变化时，优先由基础层定义不含业务语义的事件或高层接口，再由组合根显式装配适配器。

禁止用 `init()`、运行时扫描、反射注册或 build tag 隐藏模块启用状态。模块是否启用必须能从组合根的 import、Wire ProviderSet 和构造参数直接看出。

## 4. 模块接缝

| 场景 | 基础层接口 | 模块负责的实现 | 组合方式 |
| --- | --- | --- | --- |
| 用户数据变化 | `pkg/event.UserEvents`、`UserSubscriber` | 将用户变化转换为异步工作 | 在组合根中显式订阅 |
| SSE 推送 | `pkg/sse.Stream`、`Registry`、`Publisher` | 流标识、事件标识、payload 和订阅解析 | 在模块 Provider 中注册 |
| 定时任务 | `server.TaskContributor`、`pkg/job.Task` | 稳定任务名和执行器 | 在组合根中汇总注册 |
| AI 固定流程 | `service/base/agent/ai.FixedFlowProvider` | 流程、入口动作、回复和快捷入口 | 在 Provider 中注册 |
| 管理端代码生成 | `service/system/admin/codegen.ProtoTarget` | 协议、服务和前端目录元数据 | 在 Provider 中注册 |

重复的模块名、任务名、流标识、AI 流程标识或代码生成目标必须启动或生成失败，不能静默覆盖。

## 5. 新增业务模块

以下以新增 `<module>` 为例：

1. 明确模块解决的问题、拥有的表、对外协议和依赖的基础能力。
2. 在 `backend/api/proto/<module>/admin/v1` 定义接口；只有跨端共享类型才进入 `common/v1`。
3. 在 `backend/service/<module>` 实现领域规则，在 `backend/server/<module>` 实现服务注册。
4. 在 `wire_<module>.go` 集中 ProviderSet、传输服务、任务和订阅者，并在组合根启用。
5. 生成 RPC 类型后，将管理端请求放入 `frontend/admin/src/api/<module>`，页面放入 `frontend/admin/src/views/<module>`。
6. 将新增表、菜单、接口权限、字典、配置和任务写入 `sql/default-data.sql`。
7. 按协议、实现、前端和 SQL 的顺序生成并验证。

## 6. 生成与验证

```bash
cd backend
make api openapi ts wire
go test ./...
go vet ./...
make -C .. check-boundary
```

管理端代码变更执行：

```bash
cd frontend/admin
pnpm type:check
pnpm lint:oxlint
```

## 7. 删除模块

1. 删除模块的 `service`、`server`、`api/proto`、生成协议、前端页面和请求适配目录。
2. 删除对应的 Wire 文件，并从 `wire.go`、`modules.go` 和启动配置移除组合项。
3. 清理构建脚本、菜单、接口权限、配置和启用中的定时任务。
4. 执行 `go mod tidy`，重新生成协议、Wire、依赖和数据库模型。
5. 完成全量构建、测试、静态检查和文档检查。

删除不应保留临时兼容层；若基础层仍需业务类型，应先重新划分所有权。
