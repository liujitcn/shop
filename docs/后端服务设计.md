# 后端服务设计

## 文档定位

`backend` 是当前项目的服务端模块。它以 Kratos 装配 HTTP、gRPC、SSE 与 MCP 入口，维护接口协议、领域服务、GORM 数据访问、运行任务、OpenAPI 和前端静态资源托管。

## 当前分层

```mermaid
flowchart TB
  Proto["api/proto\nbase / common / system / shop"] --> Gen["api/gen/go\n生成协议代码"]
  Gen --> Service["service\nbase / system / shop"]
  Service --> Biz["service/*/biz\n业务用例"]
  Biz --> Query["pkg/gen\nmodels / query / data"]
  Query --> DB[("MySQL")]
  Biz --> Integrations["推荐、支付、队列、AI、OSS"]
  Server["server + internal/cmd/server"] --> Service
```

| 目录 | 职责 |
| --- | --- |
| `api/proto` | 按公共、系统、商城和终端划分的 Proto 契约。 |
| `api/gen/go` | Go、HTTP、gRPC、Agent Tool、MCP Tool 生成代码。 |
| `service/base` | 登录、OAuth、文件、SSE、AI 会话和 MCP 等公共能力。 |
| `service/system` | 用户、角色、菜单、租户、配置、任务、日志、代码生成等系统能力。 |
| `service/shop` | 商品、订单、评价、支付、门店、统计、推荐和商城配置。 |
| `server` | 各域服务注册、HTTP/gRPC/MCP 装配和静态资源路由。 |
| `pkg` | GORM 生成代码、鉴权、任务框架、队列、公共工具与 Eino 适配层。 |
| `internal/cmd/server` | 应用启动、Wire、内嵌 OpenAPI 和模块装配入口。 |

## 域和终端划分

| 协议包 | 服务位置 | 消费端 |
| --- | --- | --- |
| `base.v1`、`common.v1` | `service/base` | 管理后台与商城端共用。 |
| `system.admin.v1` | `service/system/admin` | 系统管理后台。 |
| `system.app.v1` | `service/system/app` | 商城端的系统基础能力。 |
| `shop.admin.v1` | `service/shop/admin` | 商品、订单、评价、统计和推荐运营后台。 |
| `shop.app.v1` | `service/shop/app` | 商城端交易、会员、浏览和推荐能力。 |

HTTP 路由由 Proto 中的 `google.api.http` 映射生成。业务服务接收生成请求并调用对应 `biz` 用例；数据访问优先使用 `pkg/gen` 生成的模型、查询对象和数据层。菜单与接口权限需要同步检查 `sql/default-data.sql`，但接口元数据和 Casbin 策略由启动过程从 OpenAPI、菜单和角色关系重建。

## 生成与装配

```mermaid
flowchart LR
  Proto["api/proto 变化"] --> API["make api"]
  Proto --> OpenAPI["make openapi"]
  Proto --> TS["make ts"]
  Schema["数据库结构变化"] --> Gorm["make gorm-gen"]
  Wiring["依赖集合变化"] --> Wire["make wire"]
  API --> Service["service / biz 实现"]
  TS --> Frontend["admin / app 调用"]
```

`make gen` 顺序执行 Go、OpenAPI、TypeScript、GORM、Wire 和格式化。以下目录为生成结果，禁止手工修改：`api/gen/go`、`pkg/gen`、`internal/cmd/server/assets/openapi.yaml`、`frontend/admin/src/rpc`、`frontend/app/src/rpc`。

## 运行任务和异步链路

任务注册框架位于 `pkg/job`，实际业务任务由服务域提供。当前覆盖的主要任务包括订单/商品日统计、交易账单比对、退款结果重试、评价审核补偿、推荐主数据同步和推荐事件回放。队列、任务或定时回调不继承请求上下文，涉及租户、门店和权限口径时必须由业务数据显式确定范围。

## 外部能力

- 推荐：`service/shop/recommend` 同时维护 Gorse 与本地推荐责任链；远端未启用、失败或结果不足时回退到本地策略。
- 支付：订单交易与门店子订单分层，支付和取消围绕交易，履约和退款围绕子订单。
- AI：`service/base/agent/ai` 管理会话、消息、SSE 和工具候选；`pkg/agent/eino` 隔离 Eino 的模型、工具、回调、中间件和固定流程适配。
- MCP：服务启动时注册生成工具，并在现有 HTTP 服务上暴露 `/mcp/{terminal}`。
- 静态资源：`/shop/*` 映射本地 OSS；`backend/data` 下含 `index.html` 的一级目录会作为 SPA 挂载。

## 配置与验证

运行配置位于 `configs`：数据库使用 `data.yaml`/`data_local.yaml`，推荐和支付使用 `configs.yaml`/`configs_local.yaml`，AI 使用 `ai.yaml`/`ai_local.yaml`，认证、OSS、OAuth、服务监听和日志分别由同名配置文件维护。

文档变更只需检查 Markdown 链接与引用路径。修改后端代码或契约时，至少执行：

```bash
cd backend
go test ./...
```

再按改动范围执行 `make api`、`make openapi`、`make ts`、`make gorm-gen`、`make wire` 或 `make gen`。
