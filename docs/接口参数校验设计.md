# 接口参数校验设计(protovalidate + ProForm 联动)

## 1. 背景与目标

当前 biz 层散落大量防御性代码:

```go
if req == nil || req.GetName() == "" {
    return errorsx.InvalidArgument("部门名称不能为空")
}
```

问题:规则重复、易遗漏、与前端 ProForm 校验各写一份且容易不一致。

目标:**校验规则统一声明在 proto 中**,后端通过中间件自动拦截,前端 ProForm 的 `rules` 按本文对照表镜像同一约束,做到"一份规则、两端一致"。

技术选型:[protovalidate](https://github.com/bufbuild/protovalidate)(CEL 运行时校验,无需生成代码)。项目已具备全部依赖:

- `api/buf.yaml` 已声明 `buf.build/bufbuild/protovalidate`
- `go.mod` 已引入 `buf.build/go/protovalidate`
- Kratos v3 自带 `middleware/validate`,原生支持 protovalidate

## 2. 总体架构

```
proto 声明规则 (buf.validate)
   │
   ├── 后端:validate 中间件统一拦截 → errorsx.InvalidArgument(中文提示)
   │        biz 层只保留业务性校验(唯一性、存在性、跨表约束)
   │
   └── 前端:ProForm field.rules 按对照表镜像同一约束(中文提示与 proto 保持一致)
```

分工约定:

| 层 | 负责 | 不负责 |
| --- | --- | --- |
| proto(protovalidate) | 格式类校验:必填、长度、范围、正则、枚举 | 业务校验 |
| biz 层 | 业务校验:唯一性、存在性、状态流转、跨表约束 | 格式类校验(删除) |
| ProForm rules | 交互体验:提交前提示,与 proto 规则一致 | 作为唯一防线(后端必兜底) |

## 3. 接入步骤

### 3.1 proto 引入

```proto
import "buf/validate/validate.proto";
```

### 3.2 buf.gen.yaml 增加 managed disable

让生成代码 import 已发布的 Go 包,不重复生成:

```yaml
managed:
  disable:
    - module: 'buf.build/bufbuild/protovalidate'   # 新增此行
```

### 3.3 后端中间件(一次接入,HTTP/gRPC 全局生效)

在 `server/http.go`、`server/grpc.go` 中间件链 auth 之后追加。为了与现有 `errorsx.InvalidArgument`(reason=`INVALID_ARGUMENT`)保持一致,不直接用 kratos 自带包装,自行封装:

```go
// pkg/middleware/validate.go
package middleware

import (
    "context"
    "errors"

    "project/pkg/errorsx"

    "buf.build/go/protovalidate"
    "github.com/go-kratos/kratos/v3/middleware"
    "google.golang.org/protobuf/proto"
)

func NewValidateMiddleware() middleware.Middleware {
    return func(handler middleware.Handler) middleware.Handler {
        return func(ctx context.Context, req any) (any, error) {
            if msg, ok := req.(proto.Message); ok {
                if err := protovalidate.Validate(msg); err != nil {
                    var ve *protovalidate.ValidationError
                    if errors.As(err, &ve) && len(ve.Violations) > 0 {
                        // 取首条违规的 message(CEL 自定义规则即为中文)
                        return nil, errorsx.InvalidArgument(ve.Violations[0].Proto.GetMessage())
                    }
                    return nil, errorsx.InvalidArgument(err.Error())
                }
            }
            return handler(ctx, req)
        }
    }
}
```

```go
// server/http.go / server/grpc.go
ms = append(ms, appMiddleware.NewAuthMiddleware(...))
ms = append(ms, appMiddleware.NewValidateMiddleware()) // 新增
```

### 3.4 中文提示约定

protovalidate 标准规则(`min_len` 等)默认英文报错。**需要中文提示的字段一律用 CEL 自定义规则**,`message` 即返回给前端的文案:

```proto
string name = 4 [(buf.validate.field).cel = {
  id: "base_dept.name.required"
  message: "部门名称不能为空"
  expression: "this.size() > 0"
}];
```

`id` 命名约定:`{消息小写蛇形}.{字段}.{规则}`,如 `base_dept.name.max_len`。

## 4. 每种验证的写法与 ProForm 联动对照

以下每节给出三段:**proto 写法**(推荐 CEL 中文版)、**ProForm rules 写法**、说明。
ProForm 的 rules 写在页面的字段配置或 `rules` 对象中(`FormItemRule[]`,底层为 async-validator)。

### 4.1 必填(字符串非空)

```proto
string name = 4 [(buf.validate.field).cel = {
  id: "base_dept.name.required"
  message: "部门名称不能为空"
  expression: "this.size() > 0"
}];
```

```ts
name: [{ required: true, message: "部门名称不能为空", trigger: "blur" }]
```

> 纯英文场景可简写:`(buf.validate.field).string.min_len = 1`。

### 4.2 必填(嵌套消息,如表单对象)

```proto
message CreateBaseDeptRequest {
  BaseDeptForm base_dept = 1 [(buf.validate.field).required = true];
}
```

前端无对应项(整个表单对象必然存在);后端由此消灭 `if req == nil || req.GetBaseDept() == nil` 类防御代码。

### 4.3 必填(数值/ID 大于 0)

```proto
int64 tenant_id = 2 [(buf.validate.field).cel = {
  id: "base_dept.tenant_id.required"
  message: "所属租户不能为空"
  expression: "this > 0"
}];
```

```ts
tenant_id: [{ required: true, message: "所属租户不能为空", trigger: "change" }]
```

> 选择型组件(select / tree-select / dict)trigger 用 `change`,输入型用 `blur`。

### 4.4 必填(枚举,如状态)

```proto
.common.v1.Status status = 100 [(buf.validate.field).enum = {defined_only: true}];
```

```ts
status: [{ required: true, message: "状态不能为空", trigger: "change" }]
```

> `defined_only` 拒绝未定义的枚举值。若 0 值不合法再叠加 CEL:`expression: "this != 0"`。

### 4.5 字符串长度

```proto
string name = 4 [(buf.validate.field).cel = {
  id: "base_dept.name.length"
  message: "部门名称长度须在 1-50 个字符之间"
  expression: "this.size() >= 1 && this.size() <= 50"
}];
```

```ts
name: [
  { required: true, message: "部门名称不能为空", trigger: "blur" },
  { min: 1, max: 50, message: "部门名称长度须在 1-50 个字符之间", trigger: "blur" }
]
```

> 注意:CEL 的 `size()` 与 JS 的 `length` 都按字符计,中文不会翻倍,两端一致。

### 4.6 数值范围

```proto
int32 sort = 5 [(buf.validate.field).cel = {
  id: "base_dept.sort.range"
  message: "排序值须在 0-9999 之间"
  expression: "this >= 0 && this <= 9999"
}];
```

```ts
sort: [{ type: "number", min: 0, max: 9999, message: "排序值须在 0-9999 之间", trigger: "blur" }]
```

> ProForm 的 `input-number` 组件建议同时配 `props: { min: 0, max: 9999 }`,从交互上直接限制。

### 4.7 正则(手机号、编码等)

```proto
string phone = 6 [(buf.validate.field).cel = {
  id: "base_user.phone.pattern"
  message: "手机号格式不正确"
  expression: "this == '' || this.matches('^1[3-9][0-9]{9}$')"
}];
```

```ts
phone: [{ pattern: /^1[3-9]\d{9}$/, message: "手机号格式不正确", trigger: "blur" }]
```

> `this == '' ||` 前缀表示"可选字段,填了才校验";必填则去掉该前缀并叠加 required 规则。
> CEL `matches` 是 RE2 语法,不支持前后向断言,写正则时两端都用 RE2 兼容写法。

### 4.8 邮箱 / URL

```proto
string email = 7 [(buf.validate.field).cel = {
  id: "base_user.email.format"
  message: "邮箱格式不正确"
  expression: "this == '' || this.isEmail()"
}];

string link = 8 [(buf.validate.field).cel = {
  id: "banner.link.format"
  message: "链接格式不正确"
  expression: "this == '' || this.isUri()"
}];
```

```ts
email: [{ type: "email", message: "邮箱格式不正确", trigger: "blur" }]
link: [{ type: "url", message: "链接格式不正确", trigger: "blur" }]
```

### 4.9 取值集合(in)

```proto
int32 status = 2 [(buf.validate.field).cel = {
  id: "base_dept.status.in"
  message: "状态取值不合法"
  expression: "this in [1, 2]"
}];
```

```ts
// 选择型组件由 options 限定取值,一般无需额外规则;需要时:
status: [{ type: "enum", enum: [1, 2], message: "状态取值不合法", trigger: "change" }]
```

### 4.10 列表(repeated)

```proto
repeated int64 role_ids = 9 [
  (buf.validate.field).cel = {
    id: "base_user.role_ids.required"
    message: "请至少选择一个角色"
    expression: "this.size() > 0"
  },
  (buf.validate.field).repeated = {
    max_items: 20
    unique: true
  }
];
```

```ts
role_ids: [
  { required: true, type: "array", message: "请至少选择一个角色", trigger: "change" },
  { type: "array", max: 20, message: "最多选择 20 个角色", trigger: "change" }
]
```

> `repeated.items` 可继续对每个元素声明规则,如 `repeated.items.cel`。

### 4.11 可选字段(optional)

```proto
optional int64 parent_id = 3 [(buf.validate.field).cel = {
  id: "base_dept.parent_id.gte"
  message: "上级部门不合法"
  expression: "this >= 0"
}];
```

> proto3 `optional` 字段未设置时**跳过校验**,设置了才执行规则——即"填了才校验"语义,无需 `ignore` 配置。前端对应"非必填但填了要合法"的规则(不带 `required`)。

### 4.12 跨字段校验(message 级 CEL)

```proto
message SaveDeptRequest {
  option (buf.validate.message).cel = {
    id: "base_dept.parent_id.not_self"
    message: "上级部门不能是当前部门"
    expression: "this.id == 0 || this.parent_id == 0 || this.id != this.parent_id"
  };
  int64 id = 1;
  int64 parent_id = 2;
}
```

```ts
// 前端用自定义 validator 联动另一个字段
parent_id: [
  {
    validator: (_rule, value, callback) => {
      if (value && value === formData.id) callback(new Error("上级部门不能是当前部门"));
      else callback();
    },
    trigger: "blur"
  }
]
```

### 4.13 时间/日期字符串

项目中时间以 `string` 传输,按格式正则校验:

```proto
string start_date = 10 [(buf.validate.field).cel = {
  id: "banner.start_date.format"
  message: "日期格式须为 yyyy-MM-dd"
  expression: "this == '' || this.matches('^[0-9]{4}-[0-9]{2}-[0-9]{2}$')"
}];
```

```ts
// date-picker 组件本身产出合法格式,一般只需 required;范围联动用自定义 validator
start_date: [{ required: true, message: "开始日期不能为空", trigger: "change" }]
```

## 6. 迁移与维护约定

1. **渐进迁移**:按 proto 文件逐个改造,改一个 proto → `make api` → 删对应 biz 防御代码 → 核对前端 rules,不要求一次性全量。
2. **两端文案一致**:proto CEL `message` 与 ProForm rule `message` 使用完全相同的中文文案,便于全局搜索核对。
3. **新增接口**:格式类校验只写 proto,禁止再在 biz 层新增 `req.GetXxx() == ""` 类检查;code-gen 模板同步更新。
4. **trigger 约定**:输入型组件(input/textarea/input-number)用 `blur`,选择型(select/dict/tree-select/date-picker/radio 等)用 `change`。
5. **后端是唯一可信防线**:前端 rules 只为交互体验,任何字段约束必须先落 proto。
6. **CEL 常用函数速查**:`size()`、`matches(re2)`、`isEmail()`、`isUri()`、`isIp()`、`isHostname()`、`in [..]`、`startsWith()`、`endsWith()`、`contains()`。完整参考:https://buf.build/docs/protovalidate/schemas/standard-rules/
