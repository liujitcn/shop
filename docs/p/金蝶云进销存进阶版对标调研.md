# 金蝶云进销存进阶版对标调研

## 1. 调研定位

本文用于支撑《进销存财务系统设计》的范围扩充，重点核对金蝶精斗云云进销存高级版的仓储、采购、销售、资金和经营管理能力，并结合当前商城项目给出可落地的设计输入。

用户口述的“进阶版”，金蝶官网和当前登录账套均称为“云进销存高级版”。本文沿用用户标题，但在陈述产品事实时使用官方名称。

本文严格区分三类内容：

1. **产品事实**：来自金蝶登录态产品 UI 或金蝶官网一手资料。
2. **当前项目事实**：来自当前仓库的《进销存财务系统设计》。
3. **设计建议**：基于产品对标和当前项目边界得出的方案，不代表金蝶内部实现。

调研日期：2026-07-21。

## 3. 金蝶高级版产品事实

### 3.1 产品范围与版本识别

- 登录态账套标题为“云进销存高级版体验账套”，一级菜单包括商品、客户、购货、销货、协同、仓库、资金、资料和设置。[S1]
- 官网将云进销存定位为适用于多门店、多仓库微型商贸企业的供应链工具，覆盖采购、销售、库存和资金管理。[S2]
- 官网版本矩阵显示，调拨、盘点、序列号盘点、其他出入库、成本调整、序列号追踪、批次保质期、收付款、核销、账户转账和往来报表等核心能力并非高级版独有；高级版的明显增量集中在扫码开单、智能录单、经营助手、自助报表、客户任务和协同要货等智能与移动能力。[S2]

这意味着本项目增加仓库、调拨、批次等能力，目标是在补齐成熟进销存的基础能力，而不是照搬高级版的营销差异功能。

## 4. 当前项目基线与缺口

当前设计已经具备较好的交易事实基础：[P1]

- 区分可售量、预占量和现存量，发货才发生物理出库和销售成本。
- 通过库存账户统一处理预占、释放和物理过账。
- 采购收货、采购退货、销售出库、销售退货、盘点和报损报溢形成不可变台账。
- 使用移动加权成本，以库存总成本作为权威值。
- 已设计供应商应付、付款、退款、核销、商户结算和业务凭证。
- 已设计角色权限、租户隔离、供应商协同和对账报表。

但当前文档明确把多仓库、库位、批次、序列号、保质期和调拨列为非目标；库存账户固定在“租户 + 门店 + SKU”粒度，所有库存单、台账、盘点和预占均围绕 `tenant_store_id + sku_id`。[P1]

因此，新增这些能力不是简单增加菜单或几张业务单表，而是要把库存账户的主键和所有出入库来源改造成可承载库存位置与追溯维度的统一模型。

| 对标领域 | 当前设计 | 对标后判断 |
| --- | --- | --- |
| 仓库 | 门店直接等同库存地点 | 必须拆分门店与仓库；门店是经营/履约组织，仓库是库存地点。 |
| 库位 | 无 | 建议纳入模型并按仓库配置是否启用，避免简单业务被强制增加操作。 |
| 调拨 | 无 | 必须增加调拨单、调出、在途、调入和差异处理。 |
| 批次保质期 | 无 | 食品、药品、美妆等商品应支持批次、生产日期、到期日、预警和先进先出。 |
| 序列号 | 无 | 高价值或售后追踪商品应支持一物一码、状态流转和历史追溯。 |
| 盘点 | 门店 + SKU 乐观盘点 | 应扩到仓库、库位、批次、序列号范围，并继续保持快照与复盘机制。 |
| 成本调整 | 只有报损报溢 | 应将“数量调整”和“成本调整”分开，成本调整不改变数量。 |
| 采购销售发票 | 明确不做税务能力 | 建议增加轻量发票登记与业务单关联，仍不做申报、认证和抵扣。 |
| 资金账户 | 仅有支付方式和会计科目语义 | 应增加资金账户及不可变账户流水，支撑账户余额和转账。 |
| 结账 | 凭证导出触发期间锁定 | 应增加独立业务期间、结账检查、反结账权限和审计。 |

## 5. 本项目应采纳的完整功能范围

本节是**设计建议**，不是金蝶产品事实。建议把目标从“小型采购库存财务闭环”调整为“适合多门店、多仓库商城的完整进销存经营闭环”，但继续保留外部财务软件作为正式总账。

### 5.1 主数据

- 仓库：编码、名称、仓库类型、所属租户、可服务门店、地址、负责人、启停状态、是否允许负库存。
- 库区与库位：仓库下可选启用的二级空间，支持收货暂存区、合格品区、不良品区、退货区等用途。
- 商品仓储属性：库存管理模式、默认仓库/库位、是否批次管理、是否保质期管理、是否序列号管理、默认保质天数、出库策略、库存上下限。
- 供应商、客户、账户、结算方式、收支类别、部门、职员、项目和单据编号规则。
- 商品多单位与条码：基本单位、采购单位、销售单位及换算率，条码绑定到 SKU、包装层级或序列号。

### 5.2 采购

- 采购申请、询价、报价、比价、采购订单、采购收货/验收、采购入库、采购退货。
- 分批到货、超收/短收容差、赠品、采购费用及费用分摊。
- 以销定购、智能补货、采购历史价、最高采购价控制、订单执行与变更跟踪。
- 采购发票登记，关联采购订单、收货、退货和应付，不扩展到税务认证。
- 供应商协同：订单确认、发货登记、对账签认和退货查看。

### 5.3 销售与商城履约

- 报价、销售订单、销售出库、销售退货/换货、送货签收和客户对账。
- 商城订单自动选择履约仓，可由后台在规则允许时改仓。
- 下单预占具体仓库库存；需要批次或序列号时，可在拣货/发货阶段分配追溯单元。
- 支持部分发货、拆单发货、缺货转仓、取消剩余量和跨仓履约。
- 销售发票登记与原销售订单、出库、退货、收款关联，不等同于税控开票。
- 保留在线支付、货到付款、退款和实物退货相互独立的事实模型。

### 5.4 仓储库存

- 多仓库、可选库位、跨仓查询和按仓可用量。
- 调拨申请、调拨出库、在途库存、调拨入库、短少/破损差异和调拨撤销/红冲。
- 采购入库、销售出库、采购退货、销售退货、其他出入库、盘盈盘亏、报损报溢。
- 批次、生产日期、到期日、近效期/过期预警、先进先出或先到期先出。
- 序列号入库、在库、预占、调出、在途、调入、销售出库、退货、报损和追溯。
- 动态盘点、盲盘、复盘、按仓库/库位/分类/批次/序列号盘点。
- 数量调整与成本调整分离；组装拆卸作为独立后续阶段，不阻塞基础仓储上线。

### 5.5 成本与期间

- 默认继续使用移动加权成本；成本核算维度至少为“账簿主体 + 仓库 + SKU”，批次是否独立核算由商品或租户配置。
- 调拨本身不形成采购销售损益；同一核算主体内转移库存数量与成本，在途期间保留原成本。
- 成本调整单只改变库存成本金额，不改变数量，并记录原因、调整前后金额及财务凭证。
- 结账前检查负库存、成本待确认、未完成调拨、未确认库存单、序列号数量不一致、批次数量不一致和库存/财务对账差异。
- 反结账必须要求独立权限、原因、操作日志，并限制后续期间已结账或已导出凭证等冲突状态。

### 5.6 资金、往来和发票

- 资金账户：现金、银行、第三方支付待结算、平台结算及其他账户。
- 不可变资金流水：收款、预收、付款、退款、其他收支、账户转账和渠道结算均写入账户流水。
- 应收应付、预收预付、核销、反核销、账龄、客户/供应商对账。
- 采购销售费用单可分摊到商品成本或计入期间费用，分摊规则需留快照。
- 发票登记保存号码、日期、购销方、价税金额、附件和业务单关联；税务申报、认证、勾选和抵扣继续作为非目标。

### 5.7 报表与预警

- 库存余额、可用量、预占、在途、收发明细、进销存汇总和库存金额。
- 仓库/库位库存、库龄、呆滞库存、缺货、超储、近效期和过期预警。
- 批次跟踪、批次保质期清单、序列号跟踪和状态表。
- 调拨执行与在途明细、盘点差异、报损报溢和成本调整明细。
- 采购订单跟踪、历史价、采购付款、供应商分析。
- 销售订单跟踪、利润、排行、客户/商品/门店/仓库分析。
- 现金银行、资金流水、应收应付、账龄、对账、平台商户结算和毛利。
- 第一阶段使用固定口径报表；稳定后再增加可配置维度与自助报表，避免先建设复杂报表平台。

### 5.8 权限与审计

- 功能权限：菜单、页面、按钮和接口权限。
- 数据权限：租户、门店、仓库、供应商、客户、部门和账簿主体。
- 仓库权限至少区分查看、收货、上架、拣货、发货、调拨、盘点、调整和成本查看。
- 采购价、销售底价、库存成本、毛利和资金账户余额分别设置敏感字段权限。
- 高风险操作采用制单与审核分离，可配置是否允许自审。
- 对审核、驳回、红冲、反核销、反结账、重算成本、批次/序列号修改记录完整操作日志。

## 6. 关键模型设计建议

本节是**本项目建议模型**，用于主设计文档后续展开，不代表金蝶表结构。

### 6.1 库存空间与账户粒度

建议的空间层次：

```text
租户
├── 门店（经营与履约组织）
└── 仓库（库存核算与作业地点）
    └── 库位（可选启用的物理位置）

门店 <-- 仓库服务门店关系 --> 仓库
```

不建议把仓库永久绑定为门店的一对一子对象。应以“仓库归属租户 + 仓库服务门店关系”表达共享仓、门店仓和中心仓。

主设计最终采用以下拆分：

- `stock_account`：仓库 + SKU 的现存、合格、预占、可售和移动加权成本权威余额。
- `stock_quant`：仓库 + 库位 + SKU + 批次 + 库存状态的实物数量明细，不单独保存成本。
- `stock_batch`：批次号、生产日期、失效日期、供应商批号和质检状态等批次身份。
- `stock_serial`：每个序列号一行当前位置和生命周期状态，并通过事件表追踪历史。
- `stock_reservation_allocation`：预占到库位、批次或序列号的拣货分配。

总量必须满足：库存账户现存量 = 实物明细汇总；批次商品的实物明细必须全部有批次，序列号商品的在库/预占序列号数必须等于对应实物数量。批次和序列号负责追溯，不默认形成独立移动平均成本账户。

### 6.2 调拨与在途

建议采用两阶段调拨，而不是确认调拨单时直接从仓库 A 移到仓库 B：

```text
草稿 -> 待审核 -> 待调出 -> 在途 -> 部分收货/已收货 -> 已关闭
                         \-> 差异处理中
```

- 调出确认：减少源仓现存量，增加调拨在途量，保留批次、序列号和成本身份。
- 调入确认：减少在途量，增加目标仓现存量；支持分批收货。
- 短少、破损或串批：生成调拨差异单，由指定责任方审核处理。
- 未调出的单据可关闭；已调出的单据不得直接删除，通过退回调拨或红冲闭环。
- 同一账簿主体内调拨不形成收入、成本或往来；跨会计主体移库不能伪装成调拨，应走内部购销或独立主体间交易。

### 6.3 批次、保质期与出库策略

- `stock_batch` 保存租户、SKU、批次号、生产日期、到期日、供应商批号等批次身份。
- 同一 SKU 是否允许重复批次号应按“租户 + SKU + 批次号”约束；日期和供应商等信息确认后不允许静默改写。
- 保质期商品必须记录到期日；生产日期和保质天数可用于计算，但计算结果应落库为快照。
- 出库建议支持人工指定、FIFO 和 FEFO；保质期商品默认 FEFO。
- 禁止过期批次销售出库；临期阈值、是否允许特批出库由租户配置并记录审批。
- 退货入库优先恢复原销售批次；无法识别原批次时进入待检或隔离库存，不直接进入可售量。

### 6.4 序列号

- `stock_serial` 至少保存 SKU、序列号、当前仓库/库位、批次、当前状态和最近来源单据。
- 状态转换必须由库存过账驱动，不允许直接编辑状态。
- 同一租户和 SKU 下序列号唯一；若业务要求全租户唯一，可进一步收紧。
- 序列号数量必须与单据行数量一致，退货时校验原销售序列号，防止串货。
- 批量扫码、导入和 PDA 操作是前端效率能力，可在基础 API 稳定后实施。

### 6.5 盘点

- 创建盘点任务时冻结范围快照，而不是全仓停业；盘点期间发生的正常出入库进入复盘判定。
- 盘点维度按商品配置展开到 SKU、批次和序列号。
- 支持明盘、盲盘、多人分区盘点和复盘，但第一阶段可先做单人动态盘点。
- 差异审核后生成盘盈/盘亏库存单，保留账面数、实盘数、差异、成本和原因。
- 盘点不允许直接覆盖库存余额。

### 6.6 成本调整与重算

- 报损报溢解决数量变化，成本调整解决金额变化，两者必须分开。
- 成本调整引用具体库存账户和期间；调增/调减金额直接写台账及凭证。
- “重算成本”首先是诊断和试算工具，输出差异报告；正式修复应生成受控调整事实，不能无审计地重写历史台账。
- 金蝶成本专题支持加权平均、移动加权和先进先出，并按仓库、商品、批次、辅助属性多维核算。[S5] 当前项目建议只把移动加权作为首期默认，待真实行业需求明确后再开放成本法配置，避免一次引入多套复杂算法。

## 7. 推荐实施顺序

### P0：库存底座

1. 仓库、仓库服务门店关系、可选库位和数据权限。
2. 库存账户粒度迁移，所有库存单和预占增加仓库维度。
3. 采购收货/入库、销售发货/出库、其他出入库和退货按仓过账。
4. 调拨出库、在途、调拨入库和差异处理。
5. 显式业务期间、结账检查、反结账和操作日志。

### P1：追溯与经营闭环

1. 批次、保质期、FEFO、近效期/过期预警。
2. 序列号状态流转、扫码出入库、序列号盘点和追溯。
3. 仓库/库位/批次/序列号盘点与复盘。
4. 成本调整单、资金账户、资金流水和账户转账。
5. 采购/销售发票登记、客户/供应商对账和账龄报表。

### P2：效率与增强

1. 智能补货、以销定购、采购价和销售限价控制。
2. 跨仓履约、部分发货、缺货转仓和移动扫码作业。
3. 组装拆卸、多单位包装条码和高级费用分摊。
4. 固定报表稳定后再增加自助报表、角色看板和智能录单。

## 8. 主设计文档同步核对结果

《进销存财务系统设计》已经按本次调研同步处理以下要点：[P1]

1. 删除“多仓库、库位、批次、序列号、保质期和调拨为非目标”的表述。
2. 把“门店即库存地点”改成门店与仓库分离，并补充仓库服务门店关系。
3. 把库存账户、预占、出入库、台账、盘点、采购收货、销售退货等全部模型补齐仓库和追溯维度。
4. 将通用 `PostMovement` 深化为预占、分配、入库、出库、调拨出入、状态转换和成本调整等业务语义接口。
5. 增加调拨、批次、序列号、成本调整、资金账户、资金流水、发票登记和业务期间模型。
6. 更新权限矩阵，加入按仓数据权限和批次/序列号/成本/结账等高风险按钮权限。
7. 更新菜单、报表口径、迁移方案、分期、测试场景和上线门槛。
8. 删除“未来出现真实需求后再增加 `stock_location_id`”的后续扩展表述，改为本期基础能力。

## 9. 结论

当前设计的库存预占、发货出库、移动加权、不可变台账、往来核销、多租户账簿和凭证桥接方向可以保留。需要重做的是库存空间与追溯粒度：从“门店 + SKU”升级为以仓库为核心、可选库位、可叠加批次或序列号、可表达在途的库存账户体系。

对标金蝶后，建议把多仓、调拨、批次保质期、序列号、盘点、成本调整、资金账户、发票登记和独立结账纳入完整范围；智能录单、自助报表、移动协同和组装拆卸可后置。这样既补齐成熟进销存的基础能力，又避免在第一阶段复制大型 ERP 的全部复杂度。
