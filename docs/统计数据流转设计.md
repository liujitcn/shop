# 统计数据流转设计

## 文档目标

本文档说明交易、门店子订单、商品、用户、推荐事件和支付账单如何进入后台工作台、分析页和报表，并明确平台交易口径与门店经营口径的切换规则。

## 统计链路总览

```mermaid
flowchart LR
  Trade["交易支付事实\norder_trade"] --> OrderStat["订单日统计任务"]
  Order["门店子订单\norder_info/order_goods"] --> OrderStat
  Order --> GoodsStat["商品日统计任务"]
  Trade --> GoodsStat
  Event["推荐事件\n浏览/曝光/点击"] --> GoodsStat
  CollectCart["收藏/购物车"] --> GoodsStat
  Pay["支付/退款单"] --> Bill["交易账单任务"]
  OrderStat --> Admin["工作台 / 订单分析 / 订单报表"]
  GoodsStat --> Admin
  Bill --> Admin
```

## 数据范围与筛选

- 不额外维护平台汇总表。默认租户无租户/门店筛选时，订单数、支付订单数和客单价按 `order_trade` 统计，避免一笔多门店交易被重复计数。
- 默认租户选择租户/门店后切换为门店子订单口径；普通租户始终由数据库租户回调隔离，并按 `order_info` 统计，可继续使用门店下拉缩小范围。
- 金额、退款和商品件数可以从门店明细动态汇总；同一交易的门店子订单金额之和必须等于交易金额。
- 交易支付方式、支付渠道和支付状态来自 `order_trade`；门店、履约和退款状态来自 `order_info`。

## 订单日统计

`OrderStatDay` 按真实业务事实时间重算门店订单指标：

1. 物理清理目标日期旧统计结果，保证任务可重复执行。
2. 在线支付以 `order_payment.success_time` 为支付事实时间；货到付款以 `order_trade.created_at` 为支付事实时间，再展开对应门店子订单累计支付金额和商品件数。
3. 退款以成功 `order_refund.success_time` 为事实时间，同一门店子订单当天多次部分退款只计一次退款订单数。
4. 取消以 `order_cancel.created_at` 为事实时间，再展开该交易下已取消的门店子订单，不能按子订单创建日期回写取消指标。
5. 按租户、门店、支付方式和支付渠道聚合上述事实。

`order_stat_day` 只保存租户/门店明细，不写平台汇总行；默认租户查询时动态汇总当前筛选范围。

## 商品日统计

| 指标 | 来源 |
| --- | --- |
| 浏览数 | 推荐事件中的 `VIEW`。 |
| 收藏数 | 用户收藏数据。 |
| 加购数 | 购物车数据。 |
| 下单数 | `order_info` 与 `order_goods`。 |
| 支付件数 / 支付金额 | `order_goods`，并通过所属 `order_trade.status` 判断支付事实。 |

商品统计按租户、门店和商品保存，不额外写平台汇总行。聚合字段通过整数化表达式读取，避免 MySQL `SUM` 的高精度十进制结果扫描到 Go `int64` 时失败。

## 工作台、分析与报表

| 页面 / 服务 | 订单口径 | 数据来源 |
| --- | --- | --- |
| 工作台 | 默认租户无筛选按交易；筛选后及普通租户按门店子订单 | `order_trade`、`order_info` 与支付事实。 |
| 订单分析 | 全局订单数和销售额按交易；门店筛选按子订单；履约分布按子订单 | `order_trade`、`order_info` 与支付事实。 |
| 订单日报 / 月报 | 全局支付订单数按交易，门店支付金额、退款和商品件数按明细汇总 | 支付事实与 `order_stat_day`。 |
| 商品分析 / 报表 | 商品行为、下单和支付指标 | `goods_stat_day` 与商品明细。 |

同一笔多门店交易会产生多张门店子订单。平台全局“订单数”只计一张交易，选择租户/门店后的经营“订单数”按可见门店子订单计数；支付操作始终只针对一张 `order_trade`。

## 交易账单比对

`TradeBill` 下载渠道支付和退款账单，保存原始记录，并分别与 `order_payment`、`order_refund` 比对金额和状态。支付单按交易记录，退款单同时保留交易与门店子订单归属。支付账单属于平台能力，不向普通租户开放。

## 与推荐和 AI 的关系

- 推荐创建事件按商品快照发送；交易首次支付成功后遍历全部门店子订单商品发送一次支付事件。
- 推荐请求可分别携带 `trade_id` 和 `order_id`，支付成功场景用交易聚合商品，订单详情场景用当前门店商品。
- AI 助手查询使用后端返回的门店分组；支付、取消动作使用 `trade_id`，履约动作使用 `order_id`。

## 任务运行建议

- 日统计任务支持指定日期重跑；源数据回补后按影响日期重算。
- 交易账单任务保留原始账单和差异状态，便于财务与技术排查。
- 统计口径变化时同步更新后端接口、管理端文案和本文档。
