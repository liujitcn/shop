# 租户体系改造说明

## 背景与目标

本次改造将商城系统从单商城视角调整为单商城、多店铺视角。租户在本系统中只表示店铺经营主体，不是独立商城；商城配置、购物与收藏、推荐、支付账单等能力仍归属于整个商城或平台，不按租户单独维护。

租户只维护与店铺经营直接相关的数据，核心是商品、订单、评价主表以及租户内的组织权限；支付履约、账单对账和商城公共配置仍由平台维护。

`user_store` 保持原有定位，只表示用户门店信息和审核后的优惠资格，不属于租户体系，因此本次没有为 `user_store` 增加 `tenant_id`，也没有把用户门店审核逻辑改造成租户逻辑。

## 主要改动范围

- 新增租户基础表与租户管理接口。
- 后台登录按租户编码定位用户，默认租户编码为 `default`。
- 后台商品、订单等店铺经营数据增加租户字段和租户范围校验。
- 商城端商品、下单、订单、支付相关链路增加店铺归属；购物与收藏仍按用户维度处理，评价仍按商品维度处理，推荐仍按整个商城维度处理。
- Casbin 权限策略增加租户维度，`v0` 存租户编号，`v1` 存角色编号，原有资源和动作字段整体后移。
- 管理端新增租户管理页面，并在用户、角色、部门、商品、订单等页面依赖后端租户上下文收敛数据范围。
- 初始化 SQL 增加默认租户、固定 `tenant` 租户管理员角色和租户化权限策略。

## 表结构变化

### 新增表

| 表名 | 说明 | 关键字段 |
| --- | --- | --- |
| `base_tenant` | 租户信息表 | `id`、`code`、`name`、`contact_name`、`contact_phone`、`status`、`remark` |

`base_tenant.code` 使用唯一索引 `unique_base_tenant`，默认初始化一条租户：

- `id = 1`
- `code = default`
- `name = 默认租户`
- `status = 1`

### 增加 tenant_id 的表

| 模块 | 表 |
| --- | --- |
| 组织与权限 | `base_dept`、`base_role`、`base_user` |
| 商品 | `goods_info` |
| 订单 | `order_info` |
| 评价 | `comment_info` |

这些表的 `tenant_id` 均为 `not null`，默认值为 `1`，注释为“租户ID”，并增加普通索引用于租户或店铺范围查询。

以下数据不进入租户体系：

- 商品分类：`goods_category` 属于全商城通用类目，不按店铺维护。
- 商品明细：`goods_prop`、`goods_sku`、`goods_spec`、`goods_stat_day` 不单独增加 `tenant_id`，需要店铺归属时通过 `goods_info` 推导。
- 订单明细与支付履约：`order_address`、`order_cancel`、`order_goods`、`order_logistics`、`order_payment`、`order_refund`、`order_stat_day`、`pay_bill` 不单独增加 `tenant_id`，需要店铺归属时通过 `order_info` 推导；`pay_bill` 是平台账单，租户不维护。
- 商城配置：`shop_banner`、`shop_hot`、`shop_hot_goods`、`shop_hot_item`、`shop_service`，属于整个商城通用配置。
- 购物与收藏：`user_cart`、`user_collect`，只和用户有关，不按租户维护。
- 评价：`comment_info` 随商品和订单快照归属到租户；`comment_discussion`、`comment_reaction`、`comment_review`、`comment_summary`、`comment_tag` 不单独增加 `tenant_id`，需要租户口径时通过 `comment_info` 或商品归属推导。
- 推荐：`recommend_event`、`recommend_request`、`recommend_request_item`，属于整个商城推荐能力，不按租户维护。

### 唯一索引调整

唯一约束只在组织权限表和业务主表维度按实际查询与业务唯一性调整，明细表不为了租户隔离额外改唯一索引。

| 表 | 调整后的唯一约束含义 |
| --- | --- |
| `base_role` | `tenant_id + code` 唯一，同一租户内角色编码唯一 |
| `base_user` | `tenant_id + user_name` 唯一，同一租户内用户账号唯一 |
| `goods_info` | 商品归属店铺，商品编码等唯一性按商品主表业务规则处理 |
| `order_info` | 订单归属店铺，订单号仍保持原业务唯一性 |

## 初始化 SQL 变化

### `sql/default-data.sql`

- 新增 `base_tenant` 初始化，写入默认租户 `default`。
- `base_dept`、`base_role`、`base_user` 初始化数据补充 `tenant_id = 1`。
- 默认角色编号固定为 `super(1)`、`tenant(2)`、`admin(3)`、`user(4)`、`guest(5)`。
- `tenant` 是租户管理员内置角色，角色编码固定为 `tenant`，菜单只保留租户经营相关能力，不能通过角色管理修改、禁用、删除或重新分配菜单。
- 新增租户管理接口元数据：
  - `OptionBaseTenants`
  - `PageBaseTenants`
  - `GetBaseTenant`
  - `CreateBaseTenant`
  - `UpdateBaseTenant`
  - `DeleteBaseTenant`
  - `SetBaseTenantStatus`
- 新增租户管理菜单：
  - `1700` 租户管理
  - `1701` 新增租户
  - `1702` 删除租户
  - `1703` 编辑租户
  - `1704` 修改租户状态
- 平台 `admin` 角色菜单包含租户管理 `1700-1704`。
- 租户 `tenant` 角色菜单明确不包含租户管理、菜单管理、API 管理、字典配置、系统配置、日志、定时任务、商城服务、推荐管理、支付账单和报表分析。

### `sql/casbin_rule.sql`

权限策略从旧格式：

```text
v0 = 角色编号
v1 = 资源路径
v2 = 动作
```

调整为新格式：

```text
v0 = 租户编号
v1 = 角色编号
v2 = 资源路径
v3 = 动作
v4 = 资源占位
```

默认初始化示例：

```text
p, default, admin, /admin/v1/xxx, GET
p, default, tenant, /admin/v1/xxx, GET
p, default, user, /admin/v1/xxx, GET
p, default, guest, /admin/v1/xxx, GET
```

常规策略动作使用真实 HTTP Method，例如 `GET`、`POST`、`PUT`、`DELETE`，不再用 `ANY` 初始化普通接口策略。历史库中缺少租户维度或缺少 `v4` 占位的旧策略会在启动重建内存策略时被跳过，待角色权限重建后写入新格式。

平台超级管理员的全量内存策略由后端启动时动态生成，示例为：

```text
p, default, super, *, *
```

## 后端逻辑变化

### 租户公共能力

默认租户常量复用 `kratos-kit/database/gorm` 中的默认租户定义：

- `DefaultTenantID = 1`
- `DefaultTenantCode = "default"`
- Casbin 策略字段中的默认租户编号。
- 登录与后台管理链路中的默认租户编码。

后续权限判断不再把租户和角色拼成一个主体字符串，而是分别使用租户编号与角色编号匹配 Casbin 策略。

### 登录与认证

- 后台登录请求增加 `tenant_code`。
- 未传 `tenant_code` 时使用默认租户 `default`。
- 登录时先通过租户编码查询 `base_tenant`，再用 `tenant_id + user_name` 查询用户。
- 禁用租户不能登录。
- Token 中继续保留真实角色编号，并补充或携带租户编号；权限校验时使用 `tenant_code + role_code` 匹配 Casbin 策略。
- 小程序用户不归属租户，登录 Token 不使用租户化角色主体；租户上下文只在访问店铺商品、下单和查询店铺订单时使用。

### 租户管理

新增 `BaseTenantService` 与 `BaseTenantCase`：

- 查询租户下拉选项。
- 查询租户分页列表。
- 查询租户详情。
- 创建租户。
- 更新租户。
- 删除租户。
- 设置租户状态。

核心保护规则：

- 只有默认租户下的 `super` 或 `admin` 具备平台管理视角。
- 创建、更新、删除、设置租户状态需要平台管理权限。
- 新建租户后自动创建默认部门、`tenant` 租户管理员角色和 `admin` 管理员账号，默认密码为 `112233`。
- 新租户管理员角色使用固定菜单白名单，明确排除平台公共管理能力。
- 默认租户不能删除。
- 默认租户不能禁用。
- 默认租户编码不能修改，因为后台默认登录入口依赖 `default`。

### 用户、角色、部门

用户、角色、部门进入租户后台管理范围，用于区分不同店铺或租户的管理账号和权限：

- `base_user`、`base_role`、`base_dept` 增加 `tenant_id`。
- 分页、下拉、树查询支持 `tenant_id` 筛选。
- 平台管理员可查看全局组织权限数据；普通租户管理员只能在当前租户上下文内查看和维护。
- 普通租户管理员只能访问自己租户的用户、角色、部门数据。
- 创建或更新用户时校验用户选择的角色和部门必须属于同一租户。
- 创建或更新部门时校验上级部门必须属于同一租户。
- 创建或更新角色时写入租户归属，并按 `tenant_code + role_code` 重建 Casbin 策略。
- 删除角色时按 `tenant_code + role_code` 删除对应权限策略。
- `super` 和 `tenant` 是内置保护角色，后端禁止修改、删除、禁用和调整菜单，前端角色页面同步隐藏这些操作。

### 商品与商城配置

只有商品主表 `goods_info` 进入租户体系；商品分类、SKU、属性、规格和商城配置不进入租户体系：

- 后台商品列表、商品表单、商品选项增加租户过滤和租户名称展示。
- 商品写入时补齐租户归属。
- 商品关联数据通过 `goods_info` 推导店铺归属，不在明细表重复存储 `tenant_id`。
- `goods_category` 是全商城通用类目，不按店铺维护。
- 轮播、热门推荐、商城服务等商城配置继续作为全商城通用配置读取和维护，不按租户区分，也不出现在租户管理员菜单中。
- 商城端默认请求使用 `tenant_id = 1`，后续可扩展为多租户入口传入。

### 购物车与下单

购物车仍按用户维度维护，不作为租户数据；仅在结算和下单时根据商品所属店铺组织展示和拆单：

- 购物车项本身不以租户作为维护边界。
- 购物车列表可按商品所属店铺分组展示，便于用户理解不同店铺的商品归属。
- 从购物车删除下单商品时仍以 `user_id + goods_id + sku_code` 为用户购物车维度处理，不引入租户作为购物车归属条件。

下单逻辑按商品所属店铺拆单：

- 下单前把商品快照按 `tenant_id` 分组。
- 每个店铺生成一张订单。
- 只有 `order_info` 写入店铺归属；`order_goods`、`order_address` 等明细数据通过订单主表推导店铺归属。
- 下单响应保留兼容字段 `order_id`，同时新增 `order_ids` 和 `orders`，返回每个店铺生成的订单。
- 确认订单、再次购买、购物车结算返回店铺分组商品和店铺维度汇总。

### 订单、支付、评价、推荐

- 后台订单列表、订单分析和工作台订单指标按租户上下文收敛，订单和订单商品返回租户名称。
- 支付单、退款、物流、取消、订单统计等订单相关明细数据不单独补充租户归属，需要店铺归属时通过 `order_info` 推导。
- `pay_bill` 是平台账单，租户菜单不包含支付账单页面，租户工作台不展示账单异常入口。
- 评价主表随商品归属到租户，租户可处理自己商品相关评价；评价讨论等明细不单独维护租户字段，工作台待审核讨论按 `comment_info.pending_discussion_count` 汇总以继承租户过滤。
- 推荐是整个商城的推荐能力，不补充租户归属，也不按租户拆分推荐数据。

新增租户管理页面：

```text

```

页面能力：

- 租户分页查询。
- 新增租户。
- 编辑租户。
- 删除租户。
- 启用或禁用租户。
- 联系人、联系电话、状态和备注字段展示与表单维护。

### 已调整页面

以下管理端页面进入租户上下文管理范围：

- `views/base/user`
- `views/base/role`
- `views/base/dept`
- `views/goods/info`
- `views/order/info`

租户管理员菜单不包含平台公共页面，包括定时任务、API 管理、商城服务、推荐、支付账单和报表分析。工作台租户视角保留订单、商品、评价和租户经营相关入口；账单风险入口在租户视角隐藏，后端账单异常数返回 0。

## 商城端变化

新增默认店铺上下文工具：

```text
frontend/app/src/utils/tenant.ts
```

当前默认值：

```text
DEFAULT_TENANT_ID = 1
DEFAULT_TENANT_CODE = "default"
```

已调整逻辑：

- 登录请求补充 `tenant_code = "default"`。
- 搜索和单品直购商品列表请求补充 `tenant_id = 1`。
- 购物车 API、购物车页面按商品所属店铺分组展示，但购物车数据仍按用户维护。
- 创建订单页面按店铺展示商品和汇总，下单后支持多订单返回。
- 商城端相关 RPC 类型同步更新。

## 不变项和边界说明

- `user_store` 不属于租户体系，不增加 `tenant_id`。
- 用户门店审核通过后的优惠逻辑仍按用户门店信息处理，不作为租户边界。
- 默认租户 `id=1/code=default` 是兼容旧数据和默认入口的基础租户，不允许删除、禁用或修改编码。
- 平台管理视角只属于默认租户下的 `super` 和 `admin`。
- 普通店铺维护商品、订单、评价主表等经营数据，也维护自己的后台用户、角色、部门权限数据；除此之外的商城配置、购物收藏、推荐、支付账单和多数业务明细表不单独增加租户概念。
